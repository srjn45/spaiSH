# WP1.3 — Opt-in Execution Sandbox Design

**Date:** 2026-07-20
**Status:** Draft
**Security-sensitive:** yes

---

## Overview

WP1.3 adds an **opt-in, default-OFF** execution sandbox that restricts the
filesystem and network reach of subprocesses spawned by `code_exec` and
untrusted `bash`. Today both tools run child processes with the *exact* OS
privileges of the `spaid` daemon — full filesystem, network, and process access
(see the honest disclaimer in `internal/tools/code_exec.go:21`). The sandbox
narrows that blast radius on Linux via **Landlock + seccomp** (or `bwrap` when
present), while remaining a no-op on platforms that cannot enforce it.

The sandbox is **defense-in-depth layered *under* the existing permission gate**.
It is not a replacement for confirmation and never suppresses a prompt. When
disabled (the default), behavior is byte-for-byte identical to today.

---

## Design constraints (non-negotiable)

1. **Never bypass the permission gate.** The tier classifier
   (`internal/permissions`) and the confirm gate in `internal/agent/agent.go`
   (lines 221–266) run *first and unchanged*. The sandbox is applied later,
   inside `tool.Run`, only after a command has already been allowed to execute.
   Enabling the sandbox MUST NOT downgrade any tier: `code_exec` stays
   `TierDestructive` (`agent.go:324`) and `bash` keeps its classified tier. A
   sandboxed command is not "safer" from the gate's point of view — it is the
   same command, with an extra containment layer beneath the human's decision.
2. **Cross-platform build safety.** The real implementation sits behind
   `//go:build linux`. A `//go:build !linux` no-op fallback compiles and does
   nothing, so darwin and windows still build. The `Sandbox` interface is
   identical on every platform; callers never use build tags.
3. **Opt-in, default OFF.** Absent config → zero-value `SandboxConfig` →
   `Enabled() == false` → the no-op path. No new dependency is exercised, no
   subprocess argv changes.
4. **Fail closed when enabled.** If the sandbox is enabled but cannot be
   established on a platform that *should* support it (Linux with neither
   Landlock nor `bwrap` available), the tool refuses to run rather than silently
   executing unsandboxed. On non-Linux, enabling logs a prominent warning and
   falls back to the no-op (the permission gate still fully applies, so the
   security posture equals today's — never worse).

## New package: `internal/sandbox/`

A small package with a stable, platform-independent interface. The two tools
(`bash`, `code_exec`) depend only on this interface; the OS-specific mechanics
are hidden behind build tags.

```go
package sandbox

import "os/exec"

// Sandbox restricts the filesystem and network reach of a child process.
// The zero-cost Noop implementation is used when disabled or unsupported.
type Sandbox interface {
    // Wrap rewrites cmd in place so the process it launches runs under the
    // sandbox's filesystem/network restrictions. For the bwrap backend this
    // prepends the bubblewrap argv; for the native backend it re-execs through
    // the in-binary trampoline (see below). Wrap MUST be called after cmd is
    // otherwise fully configured (Dir, Env, Stdout/Stderr).
    //
    // Returns an error only when the sandbox is Enabled but cannot be
    // established — callers MUST treat that as fail-closed and abort the run.
    Wrap(cmd *exec.Cmd) error

    // Enabled reports whether restrictions are actually enforced. The Noop and
    // the non-Linux fallback return false.
    Enabled() bool
}

// New builds the platform sandbox from config. On Linux it returns the real
// implementation (selecting bwrap or native Landlock+seccomp per Config.Backend
// and availability); on other platforms it returns the Noop (logging a warning
// if cfg.Enabled). It errors only for a malformed config (e.g. a non-existent
// allow_path). It never errors merely because a backend is missing — that is
// surfaced lazily by Wrap so a disabled sandbox on a machine without bwrap still
// constructs cleanly.
func New(cfg Config) (Sandbox, error)
```

> **Interface scope.** We expose only `Wrap(cmd)`. Both `bash` and `code_exec`
> already build an `*exec.Cmd` (`bash.go:41`, `code_exec.go:106`), so wrapping
> the command is the natural, minimal seam. An in-process `Apply(ctx)` variant
> (restricting the current goroutine/thread) is explicitly **out of scope** —
> restricting the daemon's own thread would leak into unrelated work and is not
> needed for a subprocess model.

### Files & build tags

| File | Build tag | Contents |
|---|---|---|
| `internal/sandbox/sandbox.go` | *(none)* | `Config`, `Sandbox` interface, `Noop`, `New` dispatch stub, and **pure** argv/ruleset builders (see below) that are testable everywhere |
| `internal/sandbox/sandbox_linux.go` | `//go:build linux` | Real impl: `bwrap` argv wrap + native Landlock/seccomp trampoline wiring |
| `internal/sandbox/sandbox_other.go` | `//go:build !linux` | No-op `New` returning `Noop{}` (warns if `cfg.Enabled`) |
| `internal/sandbox/trampoline_linux.go` | `//go:build linux` | `ApplyRestrictions(spec)` — applies Landlock ruleset + seccomp filter to the *current* process, then is `exec`'d over by the real command |

The **argv/ruleset builder functions live in the tagless `sandbox.go`** as pure
functions (`buildBwrapArgv(cfg, cmdPath, cmdArgs) []string`,
`buildTrampolineArgs(cfg) []string`) so their logic is unit-tested on darwin CI
without needing Landlock or bubblewrap present. Only the *application* of those
specs is Linux-guarded.

### Linux implementation

Two backends, selected by `Config.Backend` (`"auto"` default):

- **`bwrap`** (preferred when on `PATH`) — `Wrap` rewrites cmd to:
  ```
  bwrap --ro-bind / /  --dev /dev  --proc /proc  --tmpfs /tmp \
        --bind <allow_path> <allow_path> ...            # writable allow-list
        [--unshare-net]                                 # when allow_network=false
        --die-with-parent -- <origPath> <origArgs...>
  ```
  Zero Go dependencies; robust, well-audited isolation.
- **native Landlock + seccomp** (fallback / `backend = "landlock"`) — because
  Landlock and seccomp restrict the *calling* thread and are inherited by
  children, and Go's `exec` offers no safe post-fork/pre-exec hook, we re-exec
  through a **trampoline**: `Wrap` rewrites cmd to invoke the daemon's own binary
  as a hidden subcommand —
  ```
  <self> __sandbox-exec --allow-path=<p> ... --net=off -- <origPath> <origArgs...>
  ```
  The `__sandbox-exec` handler (in `cmd/spaid/main.go`, dispatched before normal
  startup) calls `sandbox.ApplyRestrictions`: builds a Landlock ruleset limiting
  writable/readable paths, installs a seccomp-bpf filter that denies
  `socket`/`connect`/`bind` when networking is off (plus a small deny-list of
  dangerous syscalls), then `syscall.Exec`s the real command. The trampoline
  *only* drops privileges — it performs no gating and cannot be reached except
  via `Wrap`. `self` is resolved with `os.Executable()`.

Read-only access to system paths needed by the interpreters (`/usr`, `/bin`,
`/lib*`, the Go/Python/Node/Ruby toolchains) is allowed by default; only *writes*
are constrained. The `code_exec` throwaway temp dir (`os.MkdirTemp`,
`code_exec.go:88`) and the agent working directory are always added to the
writable set, or `code_exec`/`go run` could not function.

Candidate dependencies for the native backend:
`github.com/landlock-lsm/go-landlock` and a seccomp-bpf library
(`github.com/elastic/go-seccomp-bpf` or `seccomp/libseccomp-golang`). The
`bwrap` backend needs none. Final dependency choice is deferred to
implementation; the interface does not depend on it.

## Config changes (`spaid.toml`)

A new `[sandbox]` section, decoded into a `SandboxConfig` on `Config`
(`internal/config/config.go:11`). The zero value = disabled.

```toml
[sandbox]
enabled       = false          # master opt-in. Default OFF.
allow_network = false          # when enabled, deny network by default
allow_paths   = ["/tmp"]       # extra writable paths (cwd + code_exec tmp always allowed)
backend       = "auto"         # "auto" | "bwrap" | "landlock" | "off"

# When true, bash commands matched by [permissions].allow_commands (the user's
# explicitly-blessed prefixes) run WITHOUT the sandbox — a usability carve-out
# for trusted commands that legitimately need network/fs (e.g. "git push").
# Default false = sandbox everything, including allow-listed commands.
trust_allowlisted_commands = false
```

```go
type SandboxConfig struct {
    Enabled                  bool     `toml:"enabled"`
    AllowNetwork             bool     `toml:"allow_network"`
    AllowPaths               []string `toml:"allow_paths"`
    Backend                  string   `toml:"backend"`
    TrustAllowlistedCommands bool     `toml:"trust_allowlisted_commands"`
}
```

### Defining "untrusted bash"

`code_exec` is always wrapped when the sandbox is enabled. For `bash`,
"untrusted" is defined against the existing permission allowlist:

- **Default (`trust_allowlisted_commands = false`):** every `bash` command is
  wrapped. Strongest, secure-by-default.
- **Opt-in carve-out (`= true`):** commands matched by
  `[permissions].allow_commands` (the prefixes the user has deliberately blessed,
  see `internal/permissions/policy.go:115`) bypass the sandbox; everything else
  is wrapped. This lets a user keep `git push`/`npm install` working while still
  containing arbitrary model-generated commands.

This ties "trust" to the one signal the user already controls, and never to the
model's own judgement.

## Wiring points

The sandbox is threaded into the two tools and applied strictly inside `Run`,
after the gate.

### `internal/tools/bash.go`

```go
type Bash struct {
    Sandbox sandbox.Sandbox        // nil-safe: nil ⇒ no wrapping
    Trusted func(cmd string) bool  // nil ⇒ treat nothing as trusted
}

func (b Bash) Run(ctx context.Context, input json.RawMessage) (string, error) {
    // ... unchanged parsing ...
    cmd := exec.CommandContext(ctx, "bash", "-c", args.Command)
    cmd.Stdout, cmd.Stderr = &out, &out
    if b.Sandbox != nil && b.Sandbox.Enabled() &&
        !(b.Trusted != nil && b.Trusted(args.Command)) {
        if err := b.Sandbox.Wrap(cmd); err != nil {
            return "", fmt.Errorf("sandbox setup failed (refusing to run unsandboxed): %w", err)
        }
    }
    err := cmd.Run()
    // ... unchanged ...
}
```

### `internal/tools/code_exec.go`

```go
type CodeExec struct {
    Sandbox sandbox.Sandbox // nil-safe
}

// In Run, after building cmd (code_exec.go:106) and setting cmd.Dir:
if c.Sandbox != nil && c.Sandbox.Enabled() {
    if err := c.Sandbox.Wrap(cmd); err != nil {
        return "", fmt.Errorf("sandbox setup failed (refusing to run unsandboxed): %w", err)
    }
}
```

The `code_exec` `Description()` security disclaimer is updated to note that a
sandbox *may* be active when configured, without softening the "not a safety
boundary by default" language.

### `internal/tools/tool.go`

`DefaultRegistry()` (line 54) stays sandbox-free (nil sandbox ⇒ no-op) so its
other callers (`internal/cli/slash.go:171` spec listing, the agent default at
`agent.go:73`) are unaffected. Add a constructor that injects the sandbox:

```go
func RegistryWithSandbox(sb sandbox.Sandbox, trusted func(string) bool) *Registry {
    // identical to DefaultRegistry but constructs &Bash{Sandbox: sb, Trusted: trusted}
    // and &CodeExec{Sandbox: sb}; all other tools unchanged.
}
```

### `internal/app/app.go`

In `RunAgent` (around line 331), build the sandbox once from config and pass it
into the registry. Fail closed on construction error:

```go
sb, err := sandbox.New(a.cfg.Sandbox)
if err != nil {
    return fmt.Errorf("sandbox init: %w", err) // enabled but misconfigured ⇒ abort
}
var trusted func(string) bool
if a.cfg.Sandbox.TrustAllowlistedCommands {
    trusted = a.cfg.Policy.MatchesAllowCommand // small exported helper on Policy
}
```

`toolRegistry()` (line 370) gains the sandbox parameter and uses
`tools.RegistryWithSandbox(...)` instead of `DefaultRegistry()`.

### `internal/agent/subagent.go`

The nested `delegate` agent builds its own `tools.DefaultRegistry()`
(`subagent.go:60`). Defense-in-depth MUST extend to sub-agents, so the child
inherits the parent's `Sandbox` (and `Trusted` predicate) via
`RegistryWithSandbox`. Otherwise a delegated task would escape the sandbox.

## Order of operations (why the gate is never bypassed)

```
model emits tool call
      │
      ▼
agent.go: classify() → tier            (unchanged; sandbox does NOT alter tier)
      │
      ▼
agent.go: Policy.Decide()              (DecisionDeny blocks here — unchanged)
      │
      ▼
agent.go: confirm gate (manual mode)   (human y/N — unchanged; sandbox absent here)
      │
      ▼  (only if allowed to run)
tool.Run() → Sandbox.Wrap(cmd)         (containment applied HERE, after approval)
      │
      ▼
child process runs under Landlock/seccomp/bwrap
```

The sandbox has **zero references** in the classify/Decide/confirm path. A
regression test asserts this (below).

## Test plan

### Build-tag-safe unit tests (run on all platforms — `sandbox_test.go`, tool/config tests)

- **Config:** `[sandbox]` parses into `SandboxConfig`; absent section ⇒ zero
  value ⇒ `Enabled() == false`.
- **Disabled path:** `sandbox.New(SandboxConfig{})` returns a sandbox with
  `Enabled() == false`; `Wrap` leaves `cmd.Path`/`cmd.Args` untouched.
- **Pure builders:** `buildBwrapArgv` and `buildTrampolineArgs` produce the
  expected argv for a given config — asserted without bwrap/Landlock installed
  (these functions live in the tagless file precisely so darwin CI exercises
  them).
- **Tool wiring with a test double:** a fake `Sandbox` recording `Wrap` calls —
  - `CodeExec.Run` calls `Wrap` when enabled;
  - `Bash.Run` calls `Wrap` when enabled and *not* calls it for a command the
    `Trusted` predicate matches (the allow-list carve-out);
  - `Wrap` returning an error makes `Run` fail closed (non-nil error, command
    not executed).
- **Gate-independence regression (critical):** an `internal/agent` test with a
  mock provider + mock `confirmFn` and sandbox **enabled** asserts the confirm
  prompt still fires for `code_exec`/untrusted `bash`, and that tiers are
  unchanged (`code_exec` still `TierDestructive`). Proves the sandbox neither
  suppresses nor weakens the gate.

### Linux-guarded integration test (`sandbox_linux_test.go`, `//go:build linux`)

`t.Skip` unless Linux **and** at least one backend is available
(`bwrap` on PATH, or Landlock supported by the kernel). With the sandbox enabled
and `allow_paths = [t.TempDir()]`:

- **Filesystem deny:** `cat /etc/shadow` (or a read outside the allow-list) →
  permission denied / non-zero exit.
- **Filesystem allow:** writing a file inside an allowed path succeeds.
- **Network deny:** with `allow_network = false`, a socket connect
  (`curl http://…` or a tiny `code_exec` snippet doing `connect`) fails.
- **Network allow:** with `allow_network = true`, the same connect is permitted
  (gated behind availability; may itself `t.Skip` in an offline CI).

### Acceptance gate — the no-op fallback

**`GOOS=darwin go build ./...` (and `GOOS=windows go build ./...`) MUST
succeed.** This is the explicit acceptance test that the `//go:build !linux`
no-op compiles and that the `Sandbox` interface is genuinely identical across
platforms. It is cheap, deterministic, and belongs in CI as a required check
alongside the Linux test run.

## Out of scope

- In-process (`Apply(ctx)`) restriction of the daemon itself.
- macOS/Windows enforcement (Seatbelt/AppContainer) — the no-op fallback stands;
  enabling on those platforms warns and relies on the unchanged permission gate.
- Per-command dynamic allow-path negotiation, resource (CPU/memory) limits, and
  syscall auditing/telemetry — candidates for a later WP.
- Sandboxing tools other than `bash`/`code_exec` (file tools already run
  in-process under the daemon and are gated separately).
