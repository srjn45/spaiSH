# WP2.1 — Background / Long-Running Bash Design

**Date:** 2026-07-21
**Status:** Draft

## Goal

Let the agent (and users) launch long-running shell commands without blocking
the current turn: a `run_in_background` flag on the `bash` tool starts the
process, returns a job id immediately, and keeps capturing output in memory.
A new `/jobs` REPL slash command lets the user inspect all running and
finished background jobs.

---

## Motivation

Some commands take minutes: `go build ./...`, `npm run test`, `docker build`.
Today the agent must sit in a blocking `bash` call until they complete, tying
up the turn and the context window. Background execution frees the agent to
continue, then check progress later — the same pattern Claude Code's
`run_in_background` flag exposes.

---

## Design

### 1. `run_in_background` bash tool flag

`Schema()` gains an optional boolean field `run_in_background` (default
`false`). When `false`, behaviour is unchanged. When `true`:

1. The same permission gate runs first (classification is static, happens
   before `Run()` is called by the agent loop — no change needed there).
2. A new `Job` entry is registered in the process-global `JobRegistry` with a
   monotonically increasing integer ID.
3. The command is started with `exec.Command` (not `exec.CommandContext`) so
   that context cancellation of the agent turn does not kill the background
   process.
4. `cmd.Stdout` and `cmd.Stderr` both point to a `jobWriter` that appends
   into the job's locked output buffer.
5. A goroutine calls `cmd.Wait()` and marks the job `done` or `failed` with
   the exit code.
6. `Run()` returns immediately with a human-readable message: `[job N
   started] command: <cmd>\nUse /jobs to monitor output and status.`

The sandbox wrapper (if enabled) is applied before `cmd.Start()`, same as the
foreground path. If `Wrap` fails, the job is marked `failed` and an error is
returned — never run unsandboxed.

### 2. `JobRegistry` and `Job`

```
JobRegistry   — mutex-protected slice of *Job, sequential int ID counter
Job           — ID (string), Command, status (running/done/failed), exitCode, buf
```

All exported methods on `Job` (`StatusSnapshot`, `ExitCode`, `Output`) take
the internal mutex, making them safe to call from the REPL goroutine while
the background goroutine writes output.

`globalJobs` is a package-level singleton. `Jobs()` returns it (read by
`/jobs`). `ReplaceJobs(r)` swaps it and returns a restore function — used by
tests to avoid inter-test pollution.

`JobRegistry.List()` returns all jobs newest-first. `JobRegistry.Get(id)`
finds by string ID.

### 3. `/jobs` REPL command

`/jobs`             — table of all background jobs: id, status, command
`/jobs <id>`        — detail view: id, command, status, exit code, full output

The command is read-only; it never modifies state. It is classified at
`TierRead` in `SlashTier` and is NOT wired through `confirmSlashWrite` (like
`/tools`, `/cost`, etc. — only mutating commands go through that gate).

Tab completion includes `/jobs`. The completer, `commandDetails`, `helpText`,
and `knownCommands` are all updated in sync to pass `TestHelpStaysInSync`.

### 4. Classifier change (`internal/permissions/classifier.go`)

`/jobs` is added to `slashTiers` with `TierRead`. This lets external callers
query its tier without implying any prompting.

### 5. Cross-platform

- `exec.Cmd` (`Start`/`Wait`) is stdlib and cross-platform.
- No Linux-specific syscalls, no build tags needed.
- The sandbox wrapping path is already build-tagged in `internal/sandbox`; the
  bash tool delegates to it unchanged.

## Non-goals

- Persistent job history across process restarts (in-memory only).
- `/jobs kill <id>` (not in scope for WP2.1).
- Streaming output to the REPL in real-time (polling via `/jobs <id>` is
  sufficient for V1).
