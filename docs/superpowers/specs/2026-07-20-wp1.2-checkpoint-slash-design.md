# WP1.2 — Checkpoint/Undo + Custom Slash Commands Design

**Date:** 2026-07-20
**Status:** Proposed

---

## Overview

Two roadmap items are combined into one work package because both extend
`internal/cli/slash.go` and share the same session/CLI plumbing:

- **(A) Checkpoint / undo.** Every file-mutating tool
  (`write_file`, `edit_file`, `apply_patch`, `multi_edit`) snapshots the
  original bytes of each file it touches *before* writing. Two new slash
  commands, `/undo` and `/redo`, walk that snapshot history to restore or
  re-apply the last mutation. New and deleted files round-trip correctly.

- **(B) Custom slash commands.** Markdown files under `.spai/commands/*.md`
  become user-defined slash commands. Their body is a prompt template with
  `$ARGUMENTS` and positional `$1`, `$2`, … substitution. Expansion composes
  with the existing `SPAI.md` project-context discovery: an expanded command
  runs as a normal agent turn, so it already carries project context and is
  gated by the existing tool-permission tiers.

Neither feature changes the wire protocol. Both are additive.

## Part A — Checkpoint / Undo / Redo

### Where the snapshot happens

The four mutating tools live in `internal/tools` and their `Run(ctx, input)`
signature carries no session identity, so the checkpoint store is threaded in
through `context.Context`. The agent already holds the `*session.Session` and
working directory in its dispatch loop (`internal/agent/agent.go`, just before
`tool.Run`); it constructs a checkpointer and injects it into `ctx`.

To avoid a `tools → session` import while keeping `Run` unchanged, `tools`
defines a narrow interface and the context key; `session` supplies the
implementation; the agent wires them together (dependency inversion):

```go
// internal/tools/checkpoint.go
package tools

// Checkpointer records the pre-mutation state of files. A mutating tool calls
// Snapshot(paths...) once, before it writes, passing every path it will touch.
type Checkpointer interface {
    Snapshot(paths ...string) error
}

type ctxKey struct{}

func WithCheckpointer(ctx context.Context, c Checkpointer) context.Context
func checkpointerFrom(ctx context.Context) Checkpointer // nil when absent
```

Each mutating tool gains one line before it writes:

```go
if c := checkpointerFrom(ctx); c != nil {
    _ = c.Snapshot(affectedPaths...) // best-effort; a snapshot error never blocks the write
}
```

- `write_file` / `edit_file` snapshot the single target path.
- `multi_edit` snapshots every file it is about to change (it already computes
  that set before the write loop).
- `apply_patch` snapshots every Add/Update/Delete path in its validated plan
  (Pass 1), so the whole multi-file patch is one atomic checkpoint.

When no checkpointer is in `ctx` (one-shot `spai`, unit tests, plan mode) the
tools behave exactly as today — snapshotting is a no-op.

### On-disk layout

Snapshots live in the **project tree**, keyed by session, under
`.spai/checkpoints/<session>/`. The project root is resolved the same way as
`SPAI.md`/`.spai/settings.toml` discovery: walk up from the working directory to
the nearest ancestor containing `.git` (fall back to the working dir). Both the
agent (writer) and the REPL (`/undo`, `/redo`) resolve it identically from the
same cwd, so they agree on the location.

```
.spai/checkpoints/<session>/
  index.json                 # ordered mutation log + undo/redo cursor
  0001/
    pre/   0  1  2 …          # original bytes, one blob per snapshotted file
    post/  0  1  2 …          # mutated bytes, captured lazily at undo time
  0002/
    pre/ …
  …
```

Blobs are raw file bytes (not diffs) — simplest to reason about and correct for
binary files. Each blob index (`0`, `1`, …) corresponds positionally to a file
entry in the checkpoint's `index.json` record.

### In-session index

`index.json` is the authoritative history and the undo/redo cursor:

```jsonc
{
  "cursor": 2,                 // number of *applied* checkpoints; entries[cursor:] are undone (redoable)
  "seq": 2,                    // monotonically increasing id for the next checkpoint dir
  "entries": [
    {
      "id": "0001",
      "tool": "edit_file",
      "files": [
        { "path": "internal/foo.go", "preExisted": true,  "postExisted": true },
        { "path": "internal/new.go", "preExisted": false, "postExisted": true }
      ]
    },
    { "id": "0002", "tool": "apply_patch", "files": [ /* … */ ] }
  ]
}
```

- `entries` is append-ordered. `cursor` counts how many are currently applied.
- Snapshotting a new mutation **truncates the redo tail** (`entries[cursor:]`
  and their dirs are deleted), appends the new entry, and increments `cursor` —
  standard undo-stack semantics: a fresh edit after an undo discards the
  redoable future.
- `preExisted`/`postExisted` distinguish content-restore from create/delete.

### Undo / redo mechanics

Post-images are captured **lazily at undo time** from whatever is currently on
disk, so no post-write hook is needed and the tools stay a one-liner:

- **`Snapshot(paths…)`** (before mutation): for each path, write its current
  bytes to `pre/<i>` (recording `preExisted=false` for a missing file), append
  the entry, truncate the redo tail, `cursor++`, persist `index.json`.

- **`Undo()`**: take `entries[cursor-1]`. For each file, read its **current**
  on-disk bytes into `post/<i>` (recording `postExisted`), then restore the
  pre-image — write `pre/<i>` back, or delete the file when `preExisted=false`.
  `cursor--`, persist.

- **`Redo()`**: take `entries[cursor]`. For each file, restore the post-image —
  write `post/<i>` back, or delete when `postExisted=false`. `cursor++`,
  persist.

This makes **new and deleted files round-trip**:

| Mutation | pre | post | Undo | Redo |
|----------|-----|------|------|------|
| create a file | absent | present | delete it | recreate from post |
| edit a file | old bytes | new bytes | write old bytes | write new bytes |
| delete a file (`apply_patch`) | old bytes | absent | recreate from pre | delete it |

Undo when `cursor == 0` and redo when `cursor == len(entries)` are no-ops that
report "nothing to undo/redo". A file changed on disk between mutation and undo
is captured as-is into the post-image (the redo target) — documented behavior,
not an error.

### `/undo` and `/redo` slash commands

Handled directly in `internal/cli/slash.go` (like `/clear`, `/history`), using
`r.sessionID` and `r.cwd` to open the same checkpoint store the agent wrote to:

```
case "/undo":
    r.handleUndo()   // gate (see tiers) → store.Undo() → print restored paths
case "/redo":
    r.handleRedo()
```

They print the affected paths (e.g. `undid edit_file: internal/foo.go`) and are
added to `completer()`, `knownCommands`/`commandDetails`, and `helpText`
(`slash_test.go` already asserts these stay in sync).

### Retention

The checkpoint store caps history at the most recent **50** mutations (oldest
entry + its dir pruned on append). `/clear` and `session.Clear()` also remove
`.spai/checkpoints/<session>/`. `.spai/` is expected to be git-ignored; the spec
adds a note to project docs recommending `.spai/checkpoints/` be ignored.

## Part B — Custom Slash Commands

### Discovery

Add discovery to `internal/config/project.go`, mirroring `findProjectSettings`
(same upward walk, git-root-bounded):

```go
// Command is a discovered custom slash command loaded from .spai/commands/<name>.md.
type Command struct {
    Name     string // "deploy" for /deploy (file stem, lower-cased)
    Template string // raw markdown body
}

// DiscoverCommands walks up from workingDir to the git root, reading every
// .spai/commands/*.md it finds. Nearer directories win on name collision.
func DiscoverCommands(workingDir string) ([]Command, error)
```

- Filename stem → command name: `.spai/commands/deploy.md` → `/deploy`.
- Names are lower-cased; non-`.md` files ignored; missing dir is a silent no-op.
- This composes with `SPAI.md` discovery (`internal/agent/projectcontext.go`)
  and `.spai/settings.toml` discovery: same `.spai` convention, same walk
  bounds, independent of each other.

### Argument substitution

A pure helper (`internal/config`, unit-testable) expands the template against
the command's argument list:

```go
func ExpandCommand(template string, args []string) string
```

- `$ARGUMENTS` → all args joined by a single space.
- `$1`, `$2`, … → the 1-indexed positional arg; out-of-range → empty string.
- `$$` → a literal `$` (escape). No other `$…` sequence is touched, so shell
  snippets and prose survive verbatim.

Example — `.spai/commands/fix.md`:

```
Investigate issue #$1 and open a PR. Context: $ARGUMENTS
```

`/fix 42 flaky auth test` expands to
`Investigate issue #42 and open a PR. Context: 42 flaky auth test`.

### Composition and precedence

In `handleSlash`, after the built-in `switch` falls through to `default`:

1. Strip the leading `/`, split into `name` + `args`.
2. Look the name up among discovered commands (loaded once per REPL, refreshable
   on `/init`). **Built-ins always win** — a custom `deploy.md` cannot shadow
   `/undo`, `/help`, etc.
3. On a match, `ExpandCommand` the template and submit the result through the
   normal turn path (`r.runTurn`) — identical to the user typing that prompt.
   Because it is an ordinary agent turn, it automatically picks up `SPAI.md`
   project context and every tool call it makes is gated by the existing
   permission tiers. No new tool surface, no new gate.
4. No match → existing `suggestCommand` typo hint.

Discovered command names are also fed to `completer()` so Tab completes them,
and listed under a "Custom commands" heading in `/help`.

## Permission-tier classification

The tier system (`internal/permissions`) already governs mutation gating. This
WP slots the new surface in explicitly:

- **`/undo`, `/redo`** rewrite files, so they classify at **`TierWrite`** and,
  in `manual` mode, prompt once before touching disk (auto/plan skip the
  prompt, matching every other Write action). Because they are REPL commands,
  not shell strings or tool calls, add a small classifier entry:

  ```go
  // internal/permissions/classifier.go
  // SlashTier returns the tier for a built-in mutating slash command, and false
  // for commands the tier system does not gate.
  func SlashTier(name string) (Tier, bool) // "/undo","/redo" → TierWrite, true
  ```

  `handleUndo`/`handleRedo` consult `SlashTier` and reuse the REPL's existing
  confirm path when it returns `TierWrite`.

- **Custom commands** perform no mutation themselves; they expand to a normal
  agent turn whose tool calls are already classified and gated by `classify()`
  in `internal/agent/agent.go`. No additional tier is introduced for them.

- *(Incidental note, not in scope to fix here:)* `apply_patch` currently misses
  the `write_file/edit_file/multi_edit` arm of `classify()` and falls to the
  `TierRead` default. Checkpointing is independent of tier, but the discrepancy
  is worth a follow-up so patched writes also confirm in manual mode.

## Test plan

**`internal/session/checkpoint_test.go`** — checkpoint round-trip:
- edit round-trip: snapshot → overwrite → `Undo` restores original bytes →
  `Redo` restores edited bytes.
- new-file round-trip: snapshot missing path → create → `Undo` deletes it →
  `Redo` recreates it with the same bytes.
- deleted-file round-trip: snapshot existing → delete → `Undo` recreates →
  `Redo` deletes.
- multi-file checkpoint (mimicking `apply_patch`): one entry, several files, all
  restored together.
- cursor/stack: `Undo` at empty stack and `Redo` at head are no-ops; a new
  snapshot after `Undo` truncates the redo tail.
- retention: >50 mutations prunes the oldest entry and its dir.

**`internal/tools/files_test.go`** — tool-level checkpoint integration:
- with a `Checkpointer` installed in `ctx` via `WithCheckpointer`, running
  `write_file`/`edit_file` writes a pre-image and leaves history such that
  `Undo` restores the original file on disk.
- with no checkpointer in `ctx`, the tools write normally and create no
  `.spai/checkpoints` dir (no-op path).

**`internal/permissions/classifier_test.go`**:
- `SlashTier("/undo")` and `SlashTier("/redo")` → `TierWrite, true`.
- an unrelated name → `_, false`.

**`internal/cli/slash_test.go`**:
- `/undo` and `/redo` appear in `completer`, `knownCommands`, `commandDetails`,
  and `helpText` (extends the existing in-sync assertions).
- `DiscoverCommands` finds `.spai/commands/*.md` in a temp project, lower-cases
  names, ignores non-`.md`, and honors nearer-dir precedence on collision.
- `ExpandCommand`: `$ARGUMENTS` join, `$1`/`$2` positional, out-of-range → empty,
  `$$` escape, untouched non-arg `$…`.
- built-in precedence: a custom command named like a built-in does not shadow it.

**`internal/config/project_test.go`** (if separate from slash tests): discovery
walk bounds (stops at git root) mirror `findProjectSettings`.

## Non-goals

- No cross-session undo, no global undo across all projects — history is keyed
  per `(project, session)`.
- No partial/hunk-level undo — the unit is one tool call (one checkpoint entry).
- No diff/patch storage — full pre/post byte blobs, capped by retention.
- No UI beyond the printed affected-path summary.

## Success criteria

A user edits files through the agent, then types `/undo` to revert the last
mutation (including files the agent created or deleted) and `/redo` to re-apply
it, gated once in manual mode. Dropping `deploy.md` into `.spai/commands/` makes
`/deploy <args>` expand its template — with `$ARGUMENTS` and `$1…` filled in —
and run as a normal, project-context-aware, permission-gated agent turn.
