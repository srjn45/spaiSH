# WP3.3 — Interactive REPL Polish

**Date:** 2026-07-21  
**Status:** Specified  
**Branch:** `wp3-3-repl-polish-spec`

---

## Problem

The interactive REPL is functional but spartan. Three rough edges hurt the
day-to-day feel:

1. **Single-line input only.** `rl.Readline()` returns exactly one line, so a
   user pasting a code block or composing a multi-paragraph prompt either sends
   it prematurely at the first newline or has to squash everything onto one
   line.
2. **Prefix-only `@file` completion.** `atPathCompletions` (slash.go) walks one
   directory and prefix-matches, so completing `@internal/cli/render.go` means
   Tab-descending every path component. There is no way to type a few
   distinctive letters and jump to a file deep in the tree.
3. **Diffs are colored, but bluntly.** `renderDiff` already colors add/remove/
   hunk lines, but there is no marker emphasis, and the color gate is
   `isTerminal(os.Stdout)` alone — it ignores the `NO_COLOR` convention, so a
   user who exports `NO_COLOR=1` on a TTY still gets ANSI.

All three are interactive niceties that must be **invisible to piped /
non-interactive use**: a script feeding the REPL on a pipe, or output captured
to a file, must behave byte-for-byte as it does today.

---

## Goals

1. **Multiline input** — let the user compose a multi-line message and submit it
   as one agent turn, via an explicit continuation marker and a fenced-block
   marker. Active only on a TTY stdin.
2. **Fuzzy `@file` completion** — typing `@frag` + Tab offers subsequence-fuzzy
   matches drawn from a bounded walk of the working tree, inserting the full
   relative path.
3. **Syntax-highlighted diffs** — keep the existing polarity colors, add marker
   emphasis, and honor `NO_COLOR` / non-TTY by falling back to a plain,
   ANSI-free unified diff. Reuses `internal/cli/diff.go` + `render.go`.

---

## Non-Goals

- Any change to `agent.go`, the config schema, or the tool registry. The diff
  data already flows to the renderer via `protocol.ConfirmRequest` (agent.go
  sets `ShowDiff`); this WP only changes **how** it is rendered and **how** input
  is assembled.
- A full code syntax highlighter. `chroma/v2` is already in the module graph
  (transitively, via glamour), so token coloring is *cheap* to reach for, but it
  is scoped here as an **optional stretch**; the required deliverable is
  language-agnostic polarity + marker emphasis using the existing ANSI palette.
- Editing the three superseded completion symbols in slash.go (see
  "Completion wiring" — they are wrapped, not modified, to respect the disjoint
  boundary).
- Rich editor features (cursor-up history editing of a multiline block, syntax
  highlighting *while typing*). readline stays the line editor.

## Disjointness

Per the WP brief this change is **disjoint**: it touches **only**
`internal/cli/repl.go`, `internal/cli/render.go`, `internal/cli/diff.go`, and
their `_test.go` files. It does not touch `slash.go` or `keys.go`; where it
needs behavior currently living in slash.go (the completer), it **wraps** the
existing exported-within-package symbols rather than editing them, so every
symbol in slash.go stays referenced and the package still compiles and lints
clean.

## Design

### 1. Multiline input (`repl.go`)

**Two additive submission markers**, evaluated in `Run()`'s read loop, **only
when `isTerminal(os.Stdin)`**:

| Trigger (on the just-read line) | Behavior |
|---|---|
| Line trimmed `== """` (fence open) | Enter fenced mode: read subsequent lines verbatim until a line trimmed `== """` (fence close). Body = the inner lines joined with `\n`. The two marker lines are consumed, not included. |
| Line ends with a single trailing `\` (continuation) | Strip the trailing `\`, keep the rest, and keep reading lines until one that does **not** end with `\`. Segments joined with `\n`. |
| Otherwise | Single line — today's behavior, unchanged. |

Fence is checked before continuation. A `\\` (escaped backslash) at end of line
is **not** a continuation (it collapses to a literal `\`), so a real trailing
backslash can still be sent.

**Assembly is a pure function** for testability:

```go
// assembleMultiline builds one message from an initial line plus a reader for
// subsequent lines. next returns (line, ok); ok=false signals EOF/interrupt.
// The bool result is `aborted` (Ctrl+C mid-block → discard, back to prompt).
func assembleMultiline(first string, next func() (string, bool)) (msg string, aborted bool)
```

`Run()` calls `assembleMultiline` with a `next` closure that (a) switches the
readline prompt to a dim continuation prompt (`continuationPrompt = "… "`), (b)
calls `rl.Readline()`, and (c) maps `readline.ErrInterrupt` → abort (discard the
whole block), `io.EOF` → close-out (submit what's accumulated). The restored
main prompt is set on return.

**Degradation.** When stdin is not a TTY, `Run()` skips `assembleMultiline`
entirely and submits each `rl.Readline()` line exactly as today. Piped input is
therefore unchanged: a `"""` or a trailing `\` on a pipe is treated as ordinary
line content, so no script's behavior shifts.

Constants added to repl.go: `fenceMarker = "\"\"\""`, `continuationPrompt`.

### 2. Fuzzy `@file` completion (`repl.go`)

**Completion wiring — wrap, don't edit.** slash.go's `replCompleter`
(constructed by `newCompleter`) routes `@`-words to the prefix matcher
`atPathCompletions` and `/`-lines to the slash `PrefixCompleter`. To add fuzzy
matching without touching slash.go, repl.go introduces a thin wrapper:

```go
type fuzzyCompleter struct {
    base  *replCompleter // the existing slash.go completer, reused as-is
    cwd   string
    files []string       // working-tree snapshot, gathered lazily on first @-Tab
    once  sync.Once
}
func newFuzzyCompleter(cwd string, custom ...string) *fuzzyCompleter // wraps newCompleter(cwd, custom...)
func (c *fuzzyCompleter) Do(line []rune, pos int) ([][]rune, int)
```

`Do` finds the whitespace-delimited word ending at the cursor (same logic as
`replCompleter.Do`). If it starts with `@`, it serves fuzzy file completions;
otherwise it delegates to `c.base.Do` — so `/command` completion and every
non-`@` case stay exactly as they are, and `atPathCompletions` remains
referenced (reachable via `base` for any path this wrapper doesn't intercept).
`Run()` changes one line: `newCompleter(...)` → `newFuzzyCompleter(...)`.

**Candidate gathering — bounded walk.** `gatherWorkingTree(cwd) []string` does a
`filepath.WalkDir` from `cwd`, returning file paths relative to `cwd` (slash-
separated, OS-portable via `filepath.ToSlash`), with these bounds so a huge or
pathological tree can't stall the prompt:

- **Skip set** (directory names never descended): `.git`, `node_modules`,
  `vendor`, `.worktrees`.
- **Hard cap** `maxWalkEntries = 10000`; the walk stops early once reached.
- Directories are included as candidates too (rendered with a trailing `/`).

The snapshot is taken **once** (lazy `sync.Once`) and cached for the REPL's
lifetime — the interactive session is short-lived, and re-walking on every Tab
is the only alternative. Documented tradeoff: files created mid-session are not
offered until restart (acceptable; `@literal/path` still works without
completion).

**Matching — subsequence fuzzy.**

```go
// fuzzyMatch reports whether every rune of pattern appears in candidate in
// order (case-insensitive), and a score where higher is better.
func fuzzyMatch(pattern, candidate string) (score int, ok bool)
```

Scoring (simple, deterministic, no allocation beyond lowercasing):
- Base: `ok` iff `pattern` is a subsequence of `candidate` (case-insensitive).
  Empty pattern matches everything (offer the whole snapshot, capped).
- Bonuses: +consecutive-run bonus for adjacent matched runes; +boundary bonus
  when a matched rune follows `/` or `.` or is at index 0 (favors matching
  filename starts and path segments); −gap penalty proportional to skipped runs.

`fuzzyFileCompletions(frag)` scores every cached candidate, drops non-matches,
sorts by score desc with ties broken by shorter path then lexicographic, and
keeps the top `maxFuzzyResults = 50`.

**Insertion behavior.** readline's `AutoCompleter.Do` contract returns
`(candidates [][]rune, length int)`: it removes the last `length` runes before
the cursor and appends the chosen candidate. Because a fuzzy match shares no
prefix with `frag`, we return **full relative paths** as candidates and
`length = utf8.RuneCountInString(frag)` (the text after `@`, up to the cursor).
Net effect: `@rndr`⇥ replaces `rndr` with `internal/cli/render.go`, leaving the
`@` in place → `@internal/cli/render.go`. Directory candidates carry a trailing
`/` so completion can continue into them; a lone match auto-inserts, multiple
matches list.

Constants: `maxWalkEntries`, `maxFuzzyResults`; `walkSkipDirs` set.

### 3. Syntax-highlighted diffs (`diff.go` + `render.go`)

The diff pipeline is unchanged in shape — `computeDiff` (pure) → `renderDiff`
(styling) → `printDiffPreview` (I/O). Three additive changes:

**a. `NO_COLOR` / non-TTY gate (`render.go`).** New helper:

```go
// colorEnabled reports whether ANSI styling should be emitted for f: it must be
// a terminal and NO_COLOR must be unset (https://no-color.org).
func colorEnabled(f *os.File) bool { return isTerminal(f) && os.Getenv("NO_COLOR") == "" }
```

`printDiffPreview` switches its color argument from `isTerminal(os.Stdout)` to
`colorEnabled(os.Stdout)`. When color is off, `renderDiff` already returns a
plain, prefix-bearing (`+`/`-`/` `/`@@`) unified diff — the exact fallback the
brief asks for. (init.go's identical call site keeps `isTerminal`; fully routing
it through `colorEnabled` would touch out-of-scope init.go and is flagged as a
one-line follow-up.)

**b. Marker emphasis (`diff.go`).** Keep the color scheme —

| Kind | Style (color on) | Style (color off) |
|---|---|---|
| addition (`+`) | green, **bold `+` marker** | plain `+line` |
| deletion (`-`) | red, **bold `-` marker** | plain `-line` |
| hunk (`@@`) | cyan (unchanged) | plain `@@ … @@` |
| context (` `) | uncolored (unchanged) | plain ` line` |

`renderDiff` keeps its `(lines []diffLine, color bool)` signature; the only
change is bolding the single leading marker rune of add/del lines so change
polarity reads at a glance without recoloring the payload text.

**c. Light syntax cue (optional stretch).** Because `chroma/v2` is already in
the module graph and `protocol.ConfirmRequest.Path` gives the filename, added/
context lines *could* be token-colored by lexer lookup on the extension, with a
graceful no-op when no lexer matches. This is scoped **optional**: it must not
regress the plain-diff fallback (chroma disabled entirely when
`colorEnabled == false`) and must not become a new *direct* heavy dependency
without justification. The required deliverable is (a) + (b); (c) ships only if
it stays a few lines over the already-vendored lexer.

## Test plan

All tests are pure/unit (no live terminal), table-driven per project convention.

**Fuzzy matcher (`repl_test.go`)**
- `TestFuzzyMatch_subsequence`: `("rndr","internal/cli/render.go")` matches;
  `("xyz","render.go")` does not; case-insensitivity (`"RENDER"` vs
  `"render.go"`); empty pattern matches.
- `TestFuzzyMatch_ranking`: given a fixed candidate set, an exact filename-start
  match outranks a scattered mid-path match; ties break shorter-then-lexical.
- `TestFuzzyFileCompletions_insertion`: candidates are full relative paths and
  the returned `length` equals the rune count of the fragment (verifies the
  replace-not-append insertion contract); directory candidates end in `/`;
  result count ≤ `maxFuzzyResults`.
- `TestGatherWorkingTree_bounds`: over a temp tree containing `.git/` and
  `node_modules/`, those dirs are absent from the snapshot; paths are
  slash-separated.

**Multiline assembly (`repl_test.go`)**
- `TestAssembleMultiline_fence`: `first="""`, next feeds `a`,`b`,`"""` → `"a\nb"`.
- `TestAssembleMultiline_continuation`: `first="a\\"`, next feeds `b\`, `c` →
  `"a\nb\nc"`; a trailing `\\` (escaped) does **not** continue.
- `TestAssembleMultiline_singleLine`: a plain first line returns it verbatim,
  `next` never called.
- `TestAssembleMultiline_abortAndEOF`: interrupt mid-block → `aborted=true`,
  empty msg; EOF mid-fence → submit accumulated body, `aborted=false`.

**Diff colorization (`diff_test.go`, extending the existing cases)**
- `TestRenderDiff_colorMarkers`: with `color=true`, output contains ANSI (`\033[`)
  and the bold code (`\033[1m`) wrapping `+`/`-` markers; add lines carry green
  (`\033[32m`), del lines red (`\033[31m`).
- `TestRenderDiff_noColorPlain` (existing `TestRenderDiffNonTTYPlain`, kept):
  `color=false` output is ANSI-free and still a valid prefixed unified diff.

**NO_COLOR gate (`render_test.go`)**
- `TestColorEnabled`: table over (isTerminal × `NO_COLOR` set/unset). Since
  `isTerminal` needs a real fd, assert the `NO_COLOR`-set branch returns false
  via `t.Setenv("NO_COLOR","1")` regardless, and document the TTY branch is
  covered by the diff-marker test using the `color bool` directly. The key
  assertion: **`NO_COLOR` set ⇒ no ANSI in the rendered preview.**

**Portability.** No OS-specific syscalls added (walk + string ops are stdlib);
the linux+darwin cross-build gate applies unchanged. `enterCbreak`/termios paths
are untouched.
