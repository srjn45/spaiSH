# Phase 1.7 — Piping + Session Management Design

**Date:** 2026-04-05
**Status:** Approved

## Goal

Give each terminal window its own isolated session context, let users manage that history explicitly, and allow shell pipelines to feed content into spai as labelled context.

---

## Motivation

The current daemon loads one `session.json` at startup and shares it across all invocations from all terminals. This means unrelated work in different shells bleeds into each other's context, and the session is lost on daemon restart. Phase 1.7 fixes this by routing each request to a named, file-backed session and adding tools to manage session history.

---

## Features

### 1. Per-shell Session Routing

Every `spai` request carries a `session_id` string. The daemon derives a file path from it (`~/.local/share/spaish/sessions/<id>.json`), loads the session, handles the request, saves, and drops it from memory. The daemon is fully stateless between requests.

**Default session ID resolution (CLI):**
1. If `--session <id>` flag is given, use that ID
2. Else if `$SPAI_SESSION_ID` is set (injected by shell init), use it
3. Else fall back to `"default"`

**Shell init integration:**
- Install script appends `export SPAI_SESSION_ID=$$` to `~/.bashrc` and/or `~/.zshrc` (detected from `$SHELL` and file presence)
- `$$` expands to the shell's PID at startup — unique per terminal window, stable for the shell's lifetime
- Idempotent: checks for presence before appending
- Uninstall script removes the line

**Named sessions:**
- `spai --session work` creates a new session named `"work"` if it doesn't exist, or resumes it if it does
- Session files: `~/.local/share/spaish/sessions/<id>.json`

### 2. Piped Stdin

When invoked as `cmd | spai "query"`, the CLI detects a non-TTY stdin, reads it, and sends the content in the request.

**Detection:** `os.Stdin.Stat()` — if `mode & os.ModeCharDevice == 0`, stdin is a pipe.

**Size cap:** Stdin is read up to 64 KB. If truncated, `[truncated]` is appended to the content.

**Prompt injection:** When `req.Stdin != ""`, the daemon injects a labelled user message immediately before the query in the AI message chain:

```
[piped input]
<content>
```

This applies to both the router (non-agent requests) and the agent loop. The query and piped context are kept as separate, clearly labelled messages.

### 3. Session Commands

Two new subcommands handled before flag parsing, like `spai llm`:

**`spai clear [--lines N]`**
- No `--lines`: removes all messages from the session
- `--lines N`: keeps only the latest N messages, removes older ones
- Sends a `"session"` request with command `"clear"` and optional `Lines int`
- Daemon responds with a one-line confirmation (e.g. `"Session cleared."` or `"Session trimmed to 10 messages."`)

**`spai compact`**
- Sends a `"session"` request with command `"compact"`
- Daemon calls the AI with the full message history, asking for a concise summary of what was worked on
- Replaces all messages with a single synthetic exchange:
  - `user`: `"[session summary request]"`
  - `assistant`: `"<summary text>"`
- Streams the summary back to the terminal so the user can see what was retained
- Preserves context at low token cost

## Architecture

### Request flow

```
spai                          spaid
 │                               │
 │── {session_id, stdin, ...} ──▶│
 │                               │── LoadByID(session_id) ──▶ sessions/<id>.json
 │                               │── handle request ──────────▶ AI / executor
 │                               │── session.Save() ──────────▶ sessions/<id>.json
 │◀── streamed responses ────────│
```

### Session storage

```
~/.local/share/spaish/
  sessions/
    default.json          # fallback when SPAI_SESSION_ID not set
    12345.json            # shell with PID 12345
    work.json             # named session from --session work
  spaid.sock
  spaid.log
```

### Protocol additions

```go
type Request struct {
    // existing fields ...
    SessionID string         `json:"session_id,omitempty"`
    Stdin     string         `json:"stdin,omitempty"`
    Session   *SessionRequest `json:"session,omitempty"`
}

type SessionRequest struct {
    Command string `json:"command"` // "clear" | "compact"
    Lines   int    `json:"lines,omitempty"`
}
```

## Error Handling

- **Unknown session ID:** `LoadByID` returns a fresh empty session (same as `LoadFrom` on missing file) — no error, just starts fresh
- **Session file corrupted:** `LoadByID` returns empty session, logs a warning
- **Stdin read error:** CLI logs to stderr and sends empty `Stdin` field — request continues without piped context
- **`compact` with empty session:** Daemon responds with `"Nothing to compact — session is empty."` without calling the AI

---

## Backwards Compatibility

- All changes are additive. If `session_id` is omitted from a request, the daemon uses `"default"` as the fallback ID
- Existing `session.json` (from pre-1.7 daemon) is not migrated automatically; it becomes unreachable under the new path scheme. Users starting fresh get a clean `default.json`
- The `query` request type (legacy, pre-agent) continues to work unchanged
