# WP2.3 — OpenAI `reasoning_effort` Passthrough Design

**Date:** 2026-07-21
**Status:** Draft

## Goal

Expose OpenAI's `reasoning_effort` parameter for OpenAI-compatible reasoning
models (o1, o3, o4-mini, …) so users can tune the effort level the model
spends on extended thinking. Anthropic already receives adaptive thinking
automatically; this brings parity for the OpenAI path.

---

## Motivation

The OpenAI chat completions API accepts a `reasoning_effort` field ("low",
"medium", "high") that controls how much thinking budget a reasoning model
uses. Without it, every OpenAI-compatible reasoning-model request uses the
provider default — the user has no knob. Since `internal/ai/anthropic.go`
already sets `ThinkingConfigAdaptive` unconditionally, the OpenAI path should
offer an equivalent opt-in.

---

## Design

### Wire format

When `reasoning_effort` is non-empty the field is included in the
`/chat/completions` request body:

```json
{
  "model": "o4-mini",
  "messages": [...],
  "stream": true,
  "reasoning_effort": "medium"
}
```

When empty (the default) the field is omitted via `json:"...,omitempty"`,
preserving the existing behaviour for non-reasoning models.

### Config

A new `reasoning_effort` key is added to the `[provider]` section of
`spaid.toml`:

```toml
[provider]
kind = "openai"
endpoint = "https://api.openai.com/v1"
model = "o4-mini"
reasoning_effort = "medium"   # "low" | "medium" | "high" | "" (default = omit)
```

The field maps to `ProviderConfig.ReasoningEffort` in
`internal/config/config.go`. An empty string (the TOML default for an absent
key) causes the field to be omitted from every request, which is the safe
default for non-reasoning models.

### Code path

1. `internal/config/config.go` — `ProviderConfig` gains `ReasoningEffort string`
   (`toml:"reasoning_effort"`).
2. `internal/ai/openai.go` — `OpenAIProvider` gains `reasoningEffort string`; a
   `WithReasoningEffort(effort string) *OpenAIProvider` builder method sets it.
   The `reqBody` struct gains `ReasoningEffort string`
   (`json:"reasoning_effort,omitempty"`), populated from `p.reasoningEffort`.
3. `internal/app/app.go` — `buildCloudProvider` and `SetModel` (openai case)
   call `.WithReasoningEffort(cfg.Provider.ReasoningEffort)` after construction.

### Why a builder method, not a new constructor parameter

Changing `NewOpenAIProvider`'s signature would require updating every test
call site in `provider_test.go` with a trailing `""`. A builder method
(`WithReasoningEffort`) keeps the constructor signature stable, is easy to
discover, and reads naturally at the call site.

---

## Invariants

- `reasoning_effort` is **never sent** when the field is empty — `omitempty`
  guarantees this. Non-reasoning models that receive an unknown field may error
  or ignore it; omitting it is always safe.
- No permission-tier change: this adds no new tool or command.
- No other providers (Anthropic, Ollama) are touched.

---

## Test plan

`internal/ai/openai_test.go` covers two cases:

1. `WithReasoningEffort("medium")` → the serialised request body includes
   `"reasoning_effort":"medium"`.
2. Default (no `WithReasoningEffort` call / empty string) → the request body
   does **not** contain a `reasoning_effort` key.
