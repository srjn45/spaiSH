# WP3.1 — Pre/Post Tool-Use Hooks Design

**Date:** 2026-07-21
**Status:** Draft
**Branch:** `wp3-1-hooks-spec`

---

## Problem

There is no way for a user to run their own shell logic *around* a tool call.
Two recurring needs go unmet today:

1. **Extra guard rails** — refuse an otherwise-permitted tool call based on a
   user-defined rule (e.g. "never let `write_file` touch anything under
   `vendor/`"), on top of the existing tier/confirm gate.
2. **Reactive automation** — run a command after a tool succeeds (e.g.
   auto-`gofmt` a file the model just edited), without the model having to
   remember to do it.

We want a small, opt-in hook layer wired *around* the existing tool-execution
loop, defaulting to zero hooks so behaviour is byte-identical to today.

---

## Goals

1. A new `internal/hooks/` package holding the hook model, matcher, and runner.
2. Two events: **`pre_tool`** (before a tool runs) and **`post_tool`** (after it
   succeeds).
3. `pre_tool` may **block**: a non-zero exit refuses the tool call, and the
   hook's stderr becomes the refusal reason surfaced to the model.
4. `post_tool` is **fire-and-observe**: it runs after a successful tool call;
   its failure is surfaced to the user but never undoes the tool.
5. Hooks target tools by **name glob** and optionally by a **matcher on the tool
   input**, documented precisely below.
6. **Zero-config default**: an absent `[[hooks]]` section means no hooks, so the
   tool loop runs exactly as it does today.
7. **OS-portable**: `os/exec` + `sh -c` only; no OS-specific syscalls, so the
   linux/arm64 + darwin/arm64 cross-build gate keeps passing with no build tags.

---

## Non-Goals

- Hooks that *mutate* the tool input or output (observe/block only, never
  rewrite).
- Hooks on non-tool lifecycle points (turn start/end, session start) — future
  events can extend the same `Event` enum.
- Auto-approval or tier downgrade of any kind (see invariant below).
- Persisting hook output into session history beyond the transient surfacing.

---

## CRITICAL permission invariant

**Hooks may only *add* restrictions; they can never remove one.** Concretely:

- The existing pipeline is unchanged and runs first, in order:
  `classify()` → `Policy.Decide()` (deny short-circuit) → tier-based
  `confirmFn` gate.
- A `pre_tool` hook fires **after** the `confirmFn` gate has already *approved*
  the call, and **before** `Registry.Run`. So a tool blocked by a hook was
  already permitted by the user, yet is still stopped — pure defense-in-depth,
  never a bypass.
- A hook **cannot** auto-approve, cannot satisfy or skip a confirmation prompt,
  and cannot change a tool's tier. `classify()` is not touched by this work.
- If the user *cancels* at the confirm prompt, the loop returns before the hook
  layer is reached, so **the hook never runs for a cancelled call** — a hook can
  only ever see calls the user already allowed. (This is an explicit test, see
  Test plan.)

Because hooks strictly narrow the allowed set, the worst a misconfigured hook
can do is refuse a tool that would otherwise have run — it can never widen
capability.

---

## SECURITY: trust boundary

A hook `command` is **arbitrary shell executed with the user's full
privileges**, run via `sh -c`. This is the same trust level as `SPAI.md`
(user-authored project instructions) and `[permissions].allow_commands`: the
config file is trusted input authored by the operator. Hooks are **not** a
sandbox and are **not** gated by the permission tiers — they are the operator's
own code. The spec documents this in the config comment and README so a hook is
never mistaken for a security boundary against the *model*; it is a tool *for*
the operator. Hook commands inherit the agent working directory and run under
the execution sandbox's environment only if the operator has one — the hook
process itself is **not** sandboxed by WP1.3 (out of scope; noted as a follow-up
in Non-Goals-adjacent risk).

## Design

### `internal/hooks/` package

```
internal/hooks/
  hooks.go       — Event, Hook, Runner, Invocation, BlockError, HookFailure, New()
  hooks_test.go  — matcher + block/allow/observe + timeout + env-delivery tests
```

Core types:

```go
type Event string

const (
    PreTool  Event = "pre_tool"
    PostTool Event = "post_tool"
)

// Invocation is the tool metadata handed to a hook. Output/IsError are set for
// post_tool only.
type Invocation struct {
    Tool    string          // e.g. "write_file", "mcp__github__create_issue"
    Input   json.RawMessage // raw tool-call JSON input
    Output  string          // post_tool: the tool's textual output
    IsError bool            // post_tool: always false (post runs only on success)
}

// Hook is one compiled, validated user hook.
type Hook struct {
    Event      Event
    namePat    string         // glob for path.Match against Invocation.Tool
    inputField string          // optional top-level JSON key to test
    inputRe    *regexp.Regexp  // optional RE2 pattern; nil = no input constraint
    Command    string          // shell, run via `sh -c`
    Timeout    time.Duration   // hard kill after this; default 30s
}

// Runner is the per-session compiled hook set. The zero value has no hooks and
// its Run* methods are no-ops, so an unset Config.Hooks reproduces legacy
// behaviour exactly (mirrors permissions.Policy's zero value).
type Runner struct {
    pre     []Hook
    post    []Hook
    workDir string // agent working directory; hook cmd.Dir
}

// New compiles+validates raw config hooks. A bad glob, bad regexp, unknown
// event, or empty command is a hard error (fail closed at startup).
func New(cfg []HookSpec, workDir string) (Runner, error)
```

Runner API consumed by the agent loop:

```go
// RunPre runs every matching pre_tool hook in config order and short-circuits
// on the first non-zero exit, returning a *BlockError (nil = allowed).
func (r Runner) RunPre(ctx context.Context, inv Invocation) *BlockError

// RunPost runs every matching post_tool hook in config order, running all of
// them regardless of individual failures, and returns any failures observed.
func (r Runner) RunPost(ctx context.Context, inv Invocation) []HookFailure
```

```go
// BlockError is returned when a pre_tool hook refuses a tool call.
type BlockError struct {
    Command  string // the hook command that blocked
    Reason   string // trimmed hook stderr (or a timeout notice)
    ExitCode int
}
func (e *BlockError) Error() string // "blocked by pre_tool hook: <Reason>"

// HookFailure records a post_tool hook that exited non-zero or timed out.
type HookFailure struct {
    Command  string
    Reason   string
    ExitCode int
}
```

### Matcher (documented precisely)

A hook matches an `Invocation` iff **both** of the following hold:

1. **Name glob**: `path.Match(hook.namePat, inv.Tool)` returns true. Uses Go's
   `path.Match` semantics — `*` matches any run of non-`/` characters, `?` one
   character, `[…]` a class. Tool names contain no `/`, so `*` matches an entire
   name. Examples: `write_file`, `*_file`, `mcp__*`, `*`. An empty/absent `match`
   is rejected at compile time (a hook must name what it targets).
2. **Input constraint** (optional): if `inputRe` is nil, this clause is vacuously
   true. Otherwise the pattern is matched (RE2, `regexp.MatchString`) against a
   *string* derived from the input:
   - If `input_field` is set, the value is the top-level JSON field of that name
     coerced to string (string as-is; numbers/bools via their JSON text; absent
     or non-scalar field ⇒ empty string, which the pattern is then tested
     against — so `input_field` naming a missing key matches only patterns that
     accept `""`).
   - If `input_field` is empty, the value is the **entire raw JSON input**
     (`string(inv.Input)`), letting a hook pattern-match anywhere in the args.

Matching never executes the command; it is pure and total (no error path), so it
is trivially unit-testable.

Env + stdin handed to the hook process (`exec.CommandContext(ctx, "sh", "-c",
Command)`, `cmd.Dir = workDir`):

| Channel | Contents |
|---|---|
| `SPAI_HOOK_EVENT` | `pre_tool` or `post_tool` |
| `SPAI_TOOL_NAME` | the tool name |
| `SPAI_TOOL_INPUT` | raw JSON input |
| `SPAI_TOOL_OUTPUT` | tool output (post_tool only; empty for pre) |
| stdin | raw JSON input (so scripts can `jq` it) |

Timeout via `context.WithTimeout`; on deadline the process is killed and the
result is treated as a **block** (pre) or a **failure** (post) with a timeout
`Reason`.

### Config schema (`[[hooks]]`)

Top-level array-of-tables, one entry per hook (mirrors `[[mcp.servers]]` /
`[[subagent.profiles]]`):

```toml
# Block any write under vendor/ even after the user approves it.
[[hooks]]
event       = "pre_tool"
match       = "write_file"      # glob on tool name
input_field = "path"            # optional: test this top-level input field…
input_match = "^vendor/"        # …against this RE2 pattern
command     = 'echo "refusing writes under vendor/" >&2; exit 1'
timeout_ms  = 5000

# Auto-gofmt a Go file after it is edited (fire-and-observe).
[[hooks]]
event   = "post_tool"
match   = "edit_file"
input_field = "path"
input_match = "\\.go$"
command = 'gofmt -w "$(jq -r .path <<< "$SPAI_TOOL_INPUT")"'
timeout_ms = 10000
```

Config struct in `internal/config/config.go`:

```go
// Config gains:
Hooks []HookSpec `toml:"hooks"`

// HookSpec is one user-configured shell hook run around tool execution. Zero
// entries (absent [[hooks]]) means no hooks, i.e. behaviour identical to today.
//
// SECURITY: command is arbitrary shell run with the user's privileges, same
// trust level as SPAI.md and [permissions].allow_commands. Hooks are the
// operator's own code, NOT a sandbox against the model, and NEVER bypass the
// permission gate — a pre_tool hook can only refuse an already-approved call.
type HookSpec struct {
    Event      string `toml:"event"`        // "pre_tool" | "post_tool"
    Match      string `toml:"match"`         // required glob on tool name
    InputField string `toml:"input_field"`   // optional top-level JSON key
    InputMatch string `toml:"input_match"`   // optional RE2 pattern
    Command    string `toml:"command"`       // required shell command
    TimeoutMS  int    `toml:"timeout_ms"`     // default 30000
}
```

Default `timeout_ms` = 30000. Validation (in `hooks.New`): `event` ∈ the two
constants; `match` non-empty and a valid `path.Match` pattern; `input_match`
compiles as RE2 when present; `command` non-empty.

## Exact wiring in `internal/agent/agent.go`

Add one field to `agent.Config`:

```go
// Hooks are user-configured shell hooks run AROUND tool execution, layered on
// top of (never in place of) the permission gate. The zero value has no hooks,
// preserving legacy behaviour. A pre_tool hook may only BLOCK an already-
// approved call; it can never auto-approve or change a tool's tier.
Hooks hooks.Runner
```

Inside `loop()`, in the per-tool-call section (currently `agent.go:244-325`),
insert two points. Line numbers reference the file as read for this spec.

**pre_tool — after the confirm gate, before `Registry.Run` (between the
`if needConfirm { … }` block ending at `agent.go:296` and the
`out, runErr := tool.Run(...)` at `agent.go:304`):**

```go
// pre_tool hooks: fire only AFTER the tier/confirm gate has approved this
// call (a cancelled call returns above and never reaches here) and BEFORE
// the tool runs. A block refuses the already-permitted tool — defense in
// depth, never a bypass. Mirrors the Policy-deny path: error result + continue.
inv := hooks.Invocation{Tool: tc.Name, Input: tc.Input}
if be := a.config.Hooks.RunPre(ctx, inv); be != nil {
    content := be.Error() // "blocked by pre_tool hook: <reason>"
    results = append(results, ai.ToolResult{ToolUseID: tc.ID, Content: content, IsError: true})
    send(ctx, ch, protocol.Response{Type: "output", Content: content + "\n"})
    continue // Registry.Run is NOT called
}
```

This sits after the verbose/plain `▶ …` print so the user still sees the
attempted call, then the block notice. (Placing it before the print is equally
valid; the spec fixes it *after confirm, before Run* — the print position is
cosmetic. Implementation should keep it immediately before `tool.Run`.)

**post_tool — after a *successful* `tool.Run`, before the next iteration.**
The tool already ran at `agent.go:304`; `isErr` is computed at `:305`. After the
`result`/image handling (around `agent.go:318`), add:

```go
// post_tool hooks: observe-only, run only when the tool SUCCEEDED. Failures
// are surfaced but never undo the tool (the result above is already recorded).
if !isErr {
    postInv := hooks.Invocation{Tool: tc.Name, Input: tc.Input, Output: out}
    for _, hf := range a.config.Hooks.RunPost(ctx, postInv) {
        send(ctx, ch, protocol.Response{Type: "output",
            Content: fmt.Sprintf("post_tool hook failed (%s): %s\n", hf.Command, hf.Reason)})
    }
}
```

No change to `classify()`, to `Policy.Decide`, to the `needConfirm` computation,
or to the `ModePlan` early-return. Plan mode never reaches tool execution, so
hooks never fire there — correct (nothing ran to hook around).

### App wiring (`internal/app/app.go`)

Build the runner alongside the policy/sandbox (around `app.go:375`) and fail
closed on a bad hook, exactly like the sandbox:

```go
hookRunner, err := hooks.New(a.cfg.Hooks, req.WorkingDir)
if err != nil {
    return fmt.Errorf("hooks init: %w", err)
}
// … then in agent.Config{ … }:
Hooks: hookRunner,
```

`childConfig()` (used by `runDelegate`) should propagate `Hooks` so delegated
sub-agents get the same hooks — defense-in-depth extends to sub-agents, matching
how `Sandbox` is inherited.

## Test plan

**`internal/hooks/hooks_test.go` (unit, real `sh`, portable):**

- **Matcher table** (`TestHook_matches`): name globs (`write_file`, `*_file`,
  `mcp__*`, `*`), event filtering, `input_field`+`input_match` (present field
  matches, absent field ⇒ `""`, whole-input match when no field). Pure, no exec.
- **Allow** (`TestRunPre_allow`): a `pre_tool` hook `command = "exit 0"` returns
  `nil` (tool permitted).
- **Block** (`TestRunPre_block`): `command = 'echo nope >&2; exit 3'` returns a
  `*BlockError` with `Reason == "nope"`, `ExitCode == 3`.
- **Short-circuit order**: two matching pre hooks; the first blocks; assert the
  second's side effect (a temp-file touch) never happens.
- **Observe** (`TestRunPost_observe`): a failing `post_tool` hook is returned as
  a `HookFailure`; a passing one is not. All matching post hooks run even if one
  fails.
- **Timeout**: `command = "sleep 5"` with `timeout_ms = 50` ⇒ block (pre) /
  failure (post) with a timeout `Reason`; process is killed.
- **Env/stdin delivery**: a hook that writes `$SPAI_TOOL_NAME` / `$SPAI_HOOK_EVENT`
  and its stdin to a temp file; assert contents.
- **Validation** (`TestNew_rejects`): unknown event, empty `match`, bad glob,
  uncompilable `input_match`, empty `command` each return an error.
- **Zero value** (`TestRunner_zeroValueNoop`): `hooks.Runner{}` — `RunPre`
  returns nil and `RunPost` returns empty for any invocation.

**`internal/agent/` behaviour tests (injected registry via `NewWithRegistry`):**

- **`TestPreToolBlock_doesNotRunTool`** — a spy `Tool` whose `Run` sets a
  `called` flag (or `t.Fatal`s). Configure a blocking `pre_tool` hook matching
  it. Assert: `called == false`, the tool result is `IsError` with the hook
  reason, and the loop continues (model can adapt). **Proves a pre_tool block
  refuses the tool WITHOUT calling `Registry.Run`.**
- **`TestConfirmStillFires_withHooks`** — a Write-tier tool + a `pre_tool` hook.
  A recording `confirmFn`. Assert `confirmFn` is invoked exactly as in the
  no-hooks case (same `ConfirmRequest`), i.e. hooks don't touch the gate. Then
  assert `classify()` returns identical tiers with hooks configured — **proves
  tiers are unchanged.**
- **`TestCancelledCall_neverRunsHook`** — `confirmFn` returns `false`. Assert the
  `pre_tool` hook's side effect never fires — **proves the hook runs strictly
  after (never bypasses) the confirm gate.**
- **`TestPostToolRunsAfterSuccess`** — a successful tool + a `post_tool` hook;
  assert the hook ran and the tool's result is unchanged (not undone). A
  `post_tool` failure is surfaced as an `output` response but the result stays
  `IsError == false`.
- **`TestNoHooks_identical`** — with empty `Config.Hooks`, the recorded response
  stream for a scripted turn is identical to the pre-WP3.1 baseline (zero-config
  byte-identity).

**Cross-build gate:** `hooks` uses only `os/exec`, `context`, `regexp`, `path`,
`encoding/json`, `strings`, `time` — no build tags, no syscalls. The existing
linux/arm64 + darwin/arm64 CI build compiles it unchanged. (`sh` is present on
both target OSes; documented as a hook runtime requirement.)
