# Phase 1.8 — History & Sessions

**Date:** 2026-04-05  
**Status:** Approved

## Overview

Phase 1.8 adds durable, human-readable session history and a session management surface to spaiOS. Every conversation is written to rotating markdown files as the source of truth. A lightweight JSON cache keeps recent context available for fast AI recall. Users can browse history in a pager, list and switch sessions, and rebuild degraded context from the full markdown record.

---

## Storage Layout

```
~/.local/share/spaios/sessions/<id>/
  cache.json          ← fast AI context: last 20 messages + summary string
  history.md          ← active write file
  history.001.md      ← rotated (oldest)
  history.002.md      ← ...

~/.local/share/spaios/pinned_session   ← plain text, one line: pinned session ID
```

### File rotation

When `history.md` exceeds **1 MB**, it is renamed to `history.NNN.md` (next available three-digit index) and a fresh `history.md` is started. There is no cap on the number of rotated files.

### Migration

`LoadByID` transparently migrates the old flat layout on first access:

```
sessions/<id>.json  →  sessions/<id>/cache.json
```

A new empty `history.md` is created alongside it. No manual steps required from the user.

---

## Session Resolution Order

```
1. --session flag
2. $SPAI_SESSION_ID env var
3. pinned_session file   ← new in 1.8
4. "default"
```

---

## Cache JSON Schema

```json
{
  "messages":   [...],                  // last 20 ai.Message entries
  "summary":    "...",                  // AI-generated digest of older history (may be empty)
  "updated_at": "2026-04-05T14:32:00Z"
}
```

When building AI context for a prompt:

1. If `summary` is non-empty, inject it first as a synthetic `assistant` message labelled `[context summary]`.
2. Append `messages`.

This gives the model soft memory of older history without token bloat.

---

## History Markdown Format

Each exchange is appended silently after every query. No user-facing output or permission prompt.

```markdown
## 2026-04-05 14:32 — user
why is nginx down

## 2026-04-05 14:32 — assistant
Checked the service status — nginx failed due to a port conflict on :80.

```bash
systemctl status nginx
ss -tlnp | grep :80
```

## 2026-04-05 14:32 — output
● nginx.service - A high performance web server - failed
...
```

Timestamps are UTC. The `output` section is only written when commands were actually executed.

## New CLI Commands

### `spai history [--session <id>]`

- **Runs:** client-side only — no daemon, no socket.
- **Behaviour:** concatenates all `history*.md` files for the session (sorted `history.001.md` → `history.002.md` → `history.md`), pipes the result through `less` (falls back to `more` if `less` is not available).
- **Flag:** `--session <id>` overrides session resolution; defaults to resolved session ID.

### `spai sessions`

- **Runs:** client-side only.
- **Behaviour:** lists all session directories under `sessions/`, one per line.

Output format:

```
  Sessions
  ────────────────────────────────────
* infra      (pinned)    12 msgs   2h ago
  work                    4 msgs   1d ago
  default                 0 msgs   3d ago
  84921      (shell)      8 msgs   5m ago
```

- `*` marks the currently pinned session.
- `(pinned)` is shown on the pinned session row.
- `(shell)` is shown on sessions whose ID is entirely numeric (PID-based auto-sessions).
- The session matching `$SPAI_SESSION_ID` is labelled `(shell)` to distinguish it from named sessions.
- Msg count is read from `cache.json`; last-modified is the mtime of `history.md` (or the directory if no history file exists yet).

### `spai sessions <id>`

- **Runs:** client-side only.
- **Behaviour:** writes `<id>` to `~/.local/share/spaios/pinned_session`.
- **Output:** `Switched to session '<id>'. Future queries will use this context.`
- If the session directory does not yet exist it is created (the session starts empty).

### `spai sessions --reset`

- **Runs:** client-side only.
- **Behaviour:** removes `pinned_session` file.
- **Output:** `Pinned session cleared. Falling back to $SPAI_SESSION_ID or 'default'.`

### `spai rebuild-context [--session <id>]`

- **Runs:** via daemon (requires AI provider).
- **Behaviour:**
  1. Daemon reads all `history*.md` files for the session (oldest → newest).
  2. Sends full text to AI with prompt: *"Summarise this conversation history concisely in one paragraph."*
  3. AI response becomes the new `summary` field in `cache.json`.
  4. The last 20 messages are re-extracted directly from the markdown files (deterministic parse, no AI involvement).
  5. Writes new `cache.json` with the fresh summary + re-extracted messages.
  6. Streams progress text to the user (same pattern as `spai compact`).
- **Use case:** AI context has drifted or been lost; user wants to restore it from the full markdown record.

### `spai clear [--lines N] [--session <id>]` (extended)

Existing behaviour is extended:

- `--lines 0` (default): deletes the entire session directory (all `history*.md` + `cache.json`).
- `--lines N`: trims `cache.json` messages to last N only. Markdown history files are **not modified** — they are append-only. Use `clear` (no `--lines`) to delete all history.

## Protocol Changes

`SessionRequest.Command` gains one new value:

| Command | Description |
|---|---|
| `"rebuild-context"` | Trigger full markdown re-summarisation |

No new fields are needed on `SessionRequest` — the session ID is already carried by `Request.SessionID`.

---

## `session` Package Changes

The `session` package is refactored around the directory layout:

| Existing | Change |
|---|---|
| `Session.path` (single file) | replaced by `Session.dir` (directory path) |
| `LoadByID` | migrates flat `.json` if found; creates dir as needed |
| `Save()` | renamed `SaveCache()` — writes `cache.json` only |
| `AddExchange` | unchanged (operates on in-memory messages) |
| `AppendHistory` | **new** — appends markdown block to `history.md`, rotates if > 1 MB |
| `Clear()` | extended — removes entire session directory |
| `Trim(n)` | operates on in-memory messages (no change) |
| `Compact(summary)` | operates on in-memory messages (no change) |
| `ReadAllHistory` | **new** — returns concatenated content of all history files oldest→newest |

`MessagesForPrompt()` is updated to prepend the summary as a synthetic message when non-empty.

---

## Error Handling

- `AppendHistory` failures are logged but never surfaced to the user (silent background write).
- `spai history` with no history files prints: `No history for session '<id>'.`
- `spai sessions <id>` with an unknown ID creates the directory silently (lazy init).
- `rebuild-context` on an empty history prints: `No history to rebuild from.`

---

## Out of Scope for 1.8

- History search / grep across sessions.
- Exporting sessions to other formats.
- Sharing or syncing sessions across machines.
