# WP2.2 — `gh` / PR Integration Tool Design

**Date:** 2026-07-21  
**Status:** Implemented  
**Files:** `internal/tools/gh.go`, `internal/tools/gh_test.go`,
`internal/tools/tool.go`, `internal/permissions/classifier.go`,
`internal/permissions/classifier_test.go`

---

## Problem

The existing `git` tool handles local repository operations but stops at the
GitHub boundary. Opening a pull request, checking CI status, or viewing PR
details all require dropping to a raw `bash` call — bypassing the structured
permission gate and forcing the model to reason about shell quoting.

## Goals

1. First-class `gh` PR lifecycle in the tool surface: view, list, create,
   merge, close, comment, and checkout PRs; plus convenience `create-branch`
   and `push` wrappers.
2. Per-subcommand permission tiering — identical design to the `git` tool —
   so reads auto-run while outward-facing mutations require Elevated approval.
3. No new runtime dependency beyond the already-expected `gh` CLI; `git` is
   also used for `create-branch` and `push`.

## Non-goals

- Full parity with `gh` CLI. A discrete allow-list of safe subcommands is
  intentional; unknown operations fall through to `bash`.
- Webhook, Actions, or Packages management.

## Subcommand Surface

| Subcommand      | Underlying command          | Tier        | Rationale                                      |
|-----------------|-----------------------------|-------------|------------------------------------------------|
| `pr-view`       | `gh pr view`                | Read        | Pure read; no side effects                     |
| `pr-list`       | `gh pr list`                | Read        | Pure read; no side effects                     |
| `pr-status`     | `gh pr status`              | Read        | Pure read; no side effects                     |
| `pr-checkout`   | `gh pr checkout`            | Write       | Modifies local working tree only               |
| `create-branch` | `git checkout -b`           | Write       | Modifies local repository only                 |
| `push`          | `git push`                  | Elevated    | Sends data to a remote; outward-facing         |
| `pr-create`     | `gh pr create`              | Elevated    | Creates a public PR on GitHub                  |
| `pr-comment`    | `gh pr comment`             | Elevated    | Posts a public comment on GitHub               |
| `pr-merge`      | `gh pr merge`               | Elevated    | Merges on the remote; hard to reverse cleanly  |
| `pr-close`      | `gh pr close`               | Elevated    | Closes on the remote; outward-facing           |

No subcommand reaches TierDestructive: the tool is intentionally scoped to
reversible GitHub-side operations (a closed PR can be reopened; a merged one
can be reverted with a commit). Force-push is left to the `git` tool which
already classifies it at TierDestructive.

## Permission Architecture

The GH tool is defence-in-depth *under* the permission gate, not a
replacement:

```
model calls tool
    → agent classifies via GHTier(subcommand)
        → Read:     runs silently
        → Write:    shows plan, confirms once
        → Elevated: explicit prompt, exact command shown
    → Run() execs the underlying binary (no shell)
```

`GHTier` is a pure offline function, identical in structure to `GitTier`.

## Bash Classifier Addition

The shell classifier (`Classify`) gains a `classifyGH` branch so that
`gh pr create ...` typed directly into bash is also gated at Elevated, and
`gh pr view` runs as Read — consistent with the tool's own tiering.

Mapping:
- `gh pr {view,list,status,checks}` → Read
- `gh pr checkout` → Write
- `gh pr {create,merge,close,comment,review,...}` → Elevated
- `gh repo view`, `gh run view/list/watch` → Read
- Everything else → Write (safe default)

## Error Handling

Failed commands (non-zero exit) are returned as content (not Go errors), so
the model sees `gh`'s message and exit code and can diagnose — identical to
the `git` tool. Binary-not-found is surfaced as a Go error with a human hint.
