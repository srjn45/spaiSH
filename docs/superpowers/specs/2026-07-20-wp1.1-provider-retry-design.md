# WP1.1 — Provider Retry / Backoff Design

**Date:** 2026-07-20
**Status:** Draft

## Goal

Give every AI provider a single, shared retry policy: on a transient failure
(HTTP 429 or 5xx) the request is retried with exponential backoff and jitter,
honouring a server-supplied `Retry-After` header, bounded by a maximum attempt
count, and abandoned immediately when the caller's context is cancelled.

---

## Motivation

There is **no retry logic anywhere** in `internal/ai` today. Every provider
surfaces the first transient error straight to the agent loop:

- `openai.go` — after `p.client.Do(httpReq)`, a non-`200` status becomes
  `fmt.Errorf("provider returned HTTP %d", resp.StatusCode)` (openai.go:154-157)
  and any transport error from `Do` is returned verbatim (openai.go:150-153).
- `ollama.go` — identical shape: non-`200` → `fmt.Errorf("ollama returned HTTP %d", ...)`
  (ollama.go:129-132), transport error returned verbatim (ollama.go:125-128).
- `anthropic.go` — uses the official SDK. `p.client.Messages.NewStreaming(ctx, params)`
  (anthropic.go:102) fails through `stream.Err()` inside the goroutine, emitted
  as an `Event{Type: "error"}` (anthropic.go:125-128). The SDK has its *own*
  retry, so today it behaves differently from the two hand-rolled HTTP paths.

The result is inconsistent behaviour across providers and zero resilience to the
routine `429 Too Many Requests` / `503` blips that hosted APIs return under load.

---

## Design

### Where retry lives: an `http.RoundTripper`

All three providers ultimately issue an HTTP request whose **response headers**
(and thus status code) are available *before* any streamed body is read:

- openai.go / ollama.go read `resp.StatusCode` right after `client.Do`, before
  the streaming goroutine starts scanning `resp.Body`.
- The Anthropic SDK issues its request through a `*http.Client` we can supply.

This makes an `http.RoundTripper` the natural, idiomatic seam. A wrapper decides
whether to retry purely from the *response status* (or a transport error), which
is known **at header time — before a single stream byte reaches the caller**. So
"never retry once a partial stream has begun" falls out for free: by the time the
provider goroutine is consuming the SSE body, the RoundTripper has already
returned a `200` and will never be re-entered for that request.

New file **`internal/ai/retry.go`**:

```go
// RetryConfig bounds the shared backoff policy. Zero values fall back to the
// defaults below via (RetryConfig).withDefaults().
type RetryConfig struct {
    MaxAttempts int           // total attempts incl. the first; default 4
    BaseDelay   time.Duration // first backoff step; default 500ms
    MaxDelay    time.Duration // cap on any single backoff; default 30s
}

// retryTransport wraps a base RoundTripper with bounded, jittered, backoff
// retries on 429 / 5xx and on transient transport errors.
type retryTransport struct {
    base http.RoundTripper // nil → http.DefaultTransport
    cfg  RetryConfig
    // rng is seeded once; jitter must not use a global source (test determinism)
}

// NewRetryClient returns an *http.Client whose Transport applies cfg. Providers
// use this instead of &http.Client{}. base may be nil for the default transport
// or a fake RoundTripper in tests.
func NewRetryClient(base http.RoundTripper, cfg RetryConfig) *http.Client
```

`RoundTrip` algorithm:

1. `attempt := 0`. Loop up to `cfg.MaxAttempts`.
2. Before every attempt after the first, rewind the request body: if
   `req.Body != nil` and `req.GetBody == nil`, **retry is impossible** — return
   the last result unretried. (`http.NewRequestWithContext` with the
   `*bytes.Reader` bodies used in openai.go:141 / ollama.go:119 sets `GetBody`
   automatically, so the JSON POSTs are always rewindable. GET probes have no
   body.)
3. Call `base.RoundTrip(req)`.
4. **Decide** via `shouldRetry(resp, err)`:
   - transport `err` that is not `ctx.Err()` and is `Temporary()`/timeout-like → retry;
   - `resp.StatusCode == 429` → retry;
   - `resp.StatusCode >= 500 && <= 599` → retry;
   - anything else (including all other 4xx) → **return as-is, no retry**.
5. If not retrying, return `(resp, err)`.
6. If retrying: drain and `Close()` the previous `resp.Body` (so the connection
   is reusable), compute the delay, and sleep **interruptibly**:
   `select { case <-time.After(delay): case <-req.Context().Done(): return ctx.Err() }`.
7. On the final attempt, return whatever the last try produced (the caller sees
   the real 429/5xx, not a synthetic error).

### Backoff + jitter

`delay(attempt)` = `min(MaxDelay, BaseDelay * 2^attempt)` then apply **full
jitter**: `rand.Int63n(step)` in `[0, step)`. A `Retry-After` header, when
present and parseable, **overrides** the computed backoff (still capped at
`MaxDelay`, with jitter suppressed — the server told us exactly when to return).

### `Retry-After` parsing — both forms

Per RFC 9110, `Retry-After` is either delta-seconds or an HTTP-date. A helper
`parseRetryAfter(h string, now time.Time) (time.Duration, bool)`:

- integer string → that many seconds (`strconv.Atoi`);
- otherwise `http.ParseTime(h)` (accepts the three HTTP-date formats) and return
  `until - now`, floored at 0;
- unparseable or absent → `(0, false)`, fall back to computed backoff.

`now` is injected as a parameter so the HTTP-date branch is deterministically
testable; production passes `time.Now()`.

### Context-cancellation aware

The request already carries `ctx` (openai.go:141, ollama.go:119,
`NewStreaming(ctx, …)`). `RoundTrip` never sleeps on a bare timer — every wait is
a `select` against `req.Context().Done()`, and `shouldRetry` treats
`ctx.Canceled` / `ctx.DeadlineExceeded` as terminal (never retried). Cancellation
mid-flight therefore returns promptly with the context error.

### Idempotency / partial-stream safety

Retries only ever occur on the **initial request send**, decided from response
headers before the body streams. Once `RoundTrip` returns a `200`, the provider
goroutine owns the body and the wrapper is out of the picture — a stream that has
started emitting deltas is never restarted. Request bodies here are pure
completion sends (no side effects) and rewindable, so re-sending is safe.

## Config knob

Add an optional shared `[retry]` section (top-level, because the policy spans the
Anthropic/OpenAI providers under `[provider]` *and* Ollama under `[local]`).

`internal/config/config.go`:

```go
type Config struct {
    // ... existing fields ...
    Retry RetryConfig `toml:"retry"`
}

// RetryConfig mirrors ai.RetryConfig on the wire. Durations are expressed as
// integer milliseconds for TOML friendliness.
type RetryConfig struct {
    MaxAttempts int `toml:"max_attempts"`  // default 4
    BaseDelayMS int `toml:"base_delay_ms"` // default 500
    MaxDelayMS  int `toml:"max_delay_ms"`  // default 30000
}
```

```toml
[retry]
max_attempts = 4
base_delay_ms = 500
max_delay_ms  = 30000
```

`app.go` converts `cfg.Retry` → `ai.RetryConfig` (ms → `time.Duration`) once and
threads it into each provider constructor. Absent/zero values resolve to the
defaults inside `ai` via `withDefaults()`, so the section is fully optional and
existing configs keep working unchanged.

## Wiring — exact call sites

The providers stop constructing bare clients and take a `RetryConfig`.

**`internal/ai/openai.go`**
- openai.go:16-21 (`OpenAIProvider` struct) — no shape change; `client` now holds
  the retry client.
- openai.go:26-33 (`NewOpenAIProvider`) — add a `retry RetryConfig` parameter;
  set `client: NewRetryClient(nil, retry)` in place of `&http.Client{}` (line 31).
- No change needed at openai.go:150 (`p.client.Do`) — retries are transparent.
  The existing openai.go:154-157 non-200 handling still runs, now only after
  retries are exhausted.

**`internal/ai/ollama.go`**
- ollama.go:16-20 (`OllamaProvider` struct) — `client` becomes the retry client.
- ollama.go:23-29 (`NewLocalProvider`) — add `retry RetryConfig`; replace
  `&http.Client{}` (line 27) with `NewRetryClient(nil, retry)`.
- ollama.go:125 (`p.client.Do`) unchanged; ollama.go:129-132 now post-retry.

**`internal/ai/anthropic.go`**
- anthropic.go:16-20 (`AnthropicProvider` struct) — unchanged.
- anthropic.go:24-33 (`NewAnthropicProvider`) and anthropic.go:38-50
  (`NewAnthropicProviderWithBase`) — add `retry RetryConfig`; pass two new SDK
  options:
  - `option.WithHTTPClient(NewRetryClient(nil, retry))` — routes SDK traffic
    through the shared transport;
  - `option.WithMaxRetries(0)` — **disable the SDK's own retry** so backoff is
    applied exactly once, by our transport, giving one uniform policy across all
    three providers. (Tradeoff noted below.)

**`internal/ai/provider.go`**
- Home for the shared type/helper if we prefer `retry.go` to stay minimal:
  the `RetryConfig` type and `NewRetryClient` may live here alongside the other
  cross-provider helpers (`CompleteText`, `streamToTextCh`), or in `retry.go`.
  Recommendation: keep them in **`retry.go`** (new file) and leave `provider.go`
  untouched except, if desired, a doc comment pointing to it. No functional call
  site in `provider.go` issues HTTP directly (`CompleteText`/`streamToTextCh`
  only drive `p.Stream`), so nothing there needs rewiring.

**Constructor callers (outside the four target files, listed for completeness):**
- `internal/app/app.go:97,99,147,153,247,259,267` — pass the converted
  `ai.RetryConfig` into each `New*Provider` call.
- `cmd/spai/init.go:41,47,62` — pass a default `ai.RetryConfig{}` (resolves to
  defaults) for the connectivity self-test.

### Tradeoff note — Anthropic SDK retry

The SDK already implements competent backoff-with-jitter that honours
`Retry-After`. Disabling it (`WithMaxRetries(0)`) in favour of our transport buys
**uniform, config-driven behaviour across all providers** and a single code path
to test, at the cost of no longer using the SDK's battle-tested implementation.
This is the recommended default. An alternative — leave the SDK to retry itself
and only wrap OpenAI/Ollama — is simpler but reintroduces the per-provider
divergence this WP exists to remove. We choose uniformity.

## Test Plan

`internal/ai/retry_test.go`, driven by a fake `http.RoundTripper` whose responses
are scripted per call so no network is touched:

```go
type fakeRT struct {
    resp []func(*http.Request) (*http.Response, error) // one per attempt
    n    int
}
```

Cases:

1. **429 + `Retry-After` (delta-seconds)** — first response `429` with
   `Retry-After: 1`, second `200`. Assert: two attempts, the honoured delay
   equals the header (not the computed backoff), final status `200`.
2. **429 + `Retry-After` (HTTP-date)** — header set to `now+2s` via the injected
   clock; assert the parsed delay ≈ 2s and that both date and integer forms route
   through `parseRetryAfter`.
3. **5xx retried** — `503` then `200`; assert two attempts and computed
   (jittered) backoff was used (no `Retry-After` present).
4. **Success after N retries** — `500, 500, 200` with `MaxAttempts: 4`; assert
   exactly three attempts and a `200` result.
5. **Context cancellation** — RoundTripper returns `503`; cancel the context
   during the backoff sleep; assert `RoundTrip` returns promptly with
   `context.Canceled` and makes **no** further attempt.
6. **Non-retryable 4xx** — `400` (and separately `401`, `404`); assert a single
   attempt and the response returned verbatim, no retry.
7. **Max-attempts exhausted** — all attempts `429`; assert exactly
   `MaxAttempts` calls and the final `429` surfaced (not a synthetic error).
8. **Non-rewindable body guard** — a request with `Body != nil, GetBody == nil`
   returning `503`; assert a single attempt (cannot safely replay).
9. **`parseRetryAfter` unit table** — integer, valid HTTP-date, empty, garbage →
   expected `(duration, ok)` pairs, with an injected `now`.

Jitter is made deterministic by seeding `retryTransport`'s RNG from a fixed value
(or injecting the source) so backoff assertions are stable.
