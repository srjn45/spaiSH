# Phase 1.6 — Agentic Loop + Modes Design

**Date:** 2026-04-04
**Status:** Approved

## Goal

Transform spai from a single-shot suggest-and-confirm tool into an autonomous agent loop: execute a command, observe its output, diagnose failures, propose fixes, and retry — all within a single `spai` invocation. Non-technical users can hand the wheel over completely; power users retain tier-based confirmation gates.

---

## Motivation

Current flow: `spai "query"` → AI suggests commands → user confirms → commands run → done. If something fails, the user must re-invoke spai manually.

Target flow: `spai "query"` → agent loop runs → commands execute → failures diagnosed → fixes attempted automatically → done (or graceful give-up with explanation).

---

## Modes

Two orthogonal settings control agent behaviour:

### Confirmation mode (default: tier-based)

| Tier | Default | `autonomous = true` |
|---|---|---|
| passthrough / read | runs silently | runs silently |
| write | prompts user | runs silently |
| elevated | prompts user | runs silently |
| destructive | prompts user | runs silently |

`autonomous = true` is set in `spaid.toml` or overridden per-invocation with `--autonomous`. Designed for non-technical users who want spai to handle everything.

### Verbosity (default: minimal)

| Event | Default | `--verbose` |
|---|---|---|
| AI explanation text | printed | printed |
| Command about to run | `▶ <cmd>` | `▶ [tier] <cmd>` |
| Command output (success) | suppressed | printed |
| Command output (failure) | printed | printed |
| Loop iteration header | suppressed | `── iteration N/max ──` |
| Final summary | one line | full AI reasoning |

`verbose = true` in config or `--verbose` / `-v` flag per invocation.

## Architecture

### Request flow

```
spai                          spaid                        AI
 │                               │                          │
 │──── agent request ───────────▶│                          │
 │                               │──── messages ───────────▶│
 │                               │◀─── plan (commands) ─────│
 │◀─── stream: text + plan ──────│                          │
 │                               │                          │
 │  [if tier gate needed]        │                          │
 │◀─── confirm_request ──────────│                          │
 │──── confirm_response ────────▶│                          │
 │                               │                          │
 │                               │── execute cmd ──▶ shell  │
 │◀─── stream: output ───────────│◀─── stdout/stderr ─────  │
 │                               │                          │
 │                               │  [feed output to AI]     │
 │                               │──── messages + output ──▶│
 │                               │◀─── next action ─────────│
 │           ... loop ...        │                          │
 │◀─── done (summary) ───────────│                          │
```

The socket connection stays open for the full agent loop. Both sides read and write on the same `net.Conn`. This enables the mid-loop confirm round-trip without opening a second connection.

### New internal package: `internal/agent`

The loop engine lives here. `spaid`'s `onAgent` handler instantiates it and streams its output. `spai` and `spaid` are kept thin; all loop logic is isolated and testable.

## Loop Engine State Machine

```
START
  │
  ▼
[Build messages: system prompt + session + user query]
  │
  ▼
[Call AI → parse text + commands]  ◀─────────────────────────┐
  │                                                           │
  ├── no commands → stream text → DONE (success)             │
  │                                                           │
  └── commands found                                          │
        │                                                     │
        ▼                                                     │
  [For each command:]                                         │
    ├── read/passthrough → execute silently                   │
    └── write/elevated/destructive                            │
          ├── autonomous=true  → execute silently             │
          └── autonomous=false → send confirm_request         │
                                   ├── approved → execute     │
                                   └── denied  → DONE (cancelled)
        │
        ▼
  [Capture exit code + output]
        │
        ├── exit 0 → feed output to AI → iteration_count++
        │            (one iteration = one AI call + all its commands)
        │            if count >= max_iterations → DONE (cap)
        │            loop ────────────────────────────────▶──┘
        │
        └── exit ≠ 0 → feed failure + output to AI
                        iteration_count++
                        if count >= max_iterations → DONE (gave up)
                        loop ────────────────────────────▶──┘
```

### AI message format per iteration

On each loop iteration, the AI receives:

```
[prior conversation messages]
[user]: Command output (exit 0):
$ <cmd>
<stdout/stderr>

What should we do next? If the goal is achieved, respond with no code block.
If something failed, diagnose and propose a fix.
```

### Termination conditions

| Condition | Outcome |
|---|---|
| AI returns no bash block | Goal achieved — print final text |
| User denies a confirmation | Cancelled — print what ran so far |
| `max_iterations` reached | Give up — print last AI response + "reached iteration limit" |
| Context cancelled (Ctrl+C) | Graceful stop — print what ran |
| Unrecoverable error | Print error, exit non-zero |

## Protocol Changes

### New types (`internal/protocol/protocol.go`)

```go
type AgentRequest struct {
    Query      string `json:"query"`
    Verbose    bool   `json:"verbose"`
    Autonomous bool   `json:"autonomous"`
}

type ConfirmRequest struct {
    Command   string `json:"command"`
    Tier      string `json:"tier"`
    Display   string `json:"display"`
    Iteration int    `json:"iteration"`
}

type ConfirmResponse struct {
    Approved bool `json:"approved"`
}
```

`Request` gets `Agent *AgentRequest` field.
`Response` gets two new `Type` values: `"confirm_request"` and `"confirm_response"`.

### Socket server

`AgentHandler` receives both `*json.Encoder` and `*json.Decoder` (not just encoder) to support the mid-loop confirm round-trip. `Serve` signature gains an `onAgent AgentHandler` parameter.

**Confirm round-trip wire format:**
1. spaid writes `Response{Type: "confirm_request", Content: <json-encoded ConfirmRequest>}` to encoder
2. spaid blocks reading from decoder expecting `Request{Type: "confirm_response", ...}` with a `ConfirmResponse` payload field
3. spai, upon receiving `confirm_request`, decodes the `ConfirmRequest` from `Content`, prompts the user, then writes back `Request{Type: "confirm_response"}` with the `ConfirmResponse` field set
4. spaid resumes the loop based on `ConfirmResponse.Approved`

## Config Changes (`spaid.toml`)

```toml
[agent]
autonomous     = false  # bypass all tier confirmation gates
max_iterations = 5      # retry cap per invocation
verbose        = false  # default output verbosity
```

---

## CLI Changes (`cmd/spai/main.go`)

- Default invocation (`spai "query"`) uses the agent flow — the old `"query"` request type is retired for interactive use
- `--verbose` / `-v` flag — verbose output for this invocation
- `--autonomous` flag — bypass confirmations for this invocation
- `--dry-run` remains — shows the first plan, never enters the loop
- Mid-stream: when `spai` receives a `confirm_request` response, it prompts the user (existing `confirmPlan` logic) and writes a `confirm_response` back on the same connection

## File Map

| File | Action |
|---|---|
| `internal/protocol/protocol.go` | Add `AgentRequest`, `ConfirmRequest`, `ConfirmResponse`; add `Agent` field to `Request` |
| `internal/agent/agent.go` | Create — loop engine, `Run()` returns `<-chan protocol.Response` |
| `internal/agent/agent_test.go` | Create — mock AI + executor tests |
| `internal/socket/server.go` | Add `AgentHandler` type; extend `Serve`; pass decoder to agent handler |
| `cmd/spaid/main.go` | Add `onAgent` closure; load `AgentConfig` from toml |
| `cmd/spai/main.go` | Default to agent flow; handle `confirm_request` mid-stream; add `--verbose`, `--autonomous` flags |
| `internal/config/config.go` | Add `AgentConfig` struct |

**Minor addition:** `internal/router` — export `SelectProvider(forceLocal bool) (ai.Provider, error)` so `spaid`'s `onAgent` handler can pass an `ai.Provider` directly to the agent engine. The agent does not use the router's `Route()` method; it manages its own message loop.

**Unchanged:** `internal/llm`, `internal/executor`, `internal/permissions`, `internal/session`, `internal/ai`

## Testing Strategy

- `internal/agent` tested with a mock `ai.Provider` and mock executor — no real shell or network calls
- Tests cover: success on first try, failure + fix, max_iterations cap, user denial mid-loop, Ctrl+C cancellation
- Existing tests for all other packages must continue to pass unchanged
- TDD: tests written before implementation for `internal/agent`

---

## Out of Scope (Spec B)

- Piping (`ls | spai "query"`)
- Session management (`spai clear`, `spai compact`, `--session` flag, per-shell session IDs)
