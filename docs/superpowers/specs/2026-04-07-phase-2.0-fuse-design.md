# Phase 2.0 — FUSE Filesystem Design

**Date:** 2026-04-07
**Status:** Approved

---

## Overview

Phase 2.0 adds a FUSE filesystem mounted at `/ai` that makes `spaid` accessible from any process without an SDK, CLI flags, or knowledge of spaiSH. Reading a virtual file triggers an AI operation and returns the result as file content.

```bash
cat /ai/explain/etc/nginx/nginx.conf       # explains the config
cat /ai/fix/etc/nginx/nginx.conf           # returns a corrected version
cat /ai/summarise/var/log/syslog           # summarises the log
cat /ai/security/etc/ssh/sshd_config       # security audit
cat "/ai/ask/what is using port 8080"      # free-form question
```

Any program that can read a file — scripts, editors, cron jobs — gets AI capabilities without knowing spaiSH exists.

---

## Scope

### Phase 2.0 — Normal user operations (this spec)

| Operation | Description |
|-----------|-------------|
| `explain` | Plain-English explanation of what a file or config does |
| `summarise` | Short 3–5 sentence summary of file content |
| `fix` | Full corrected version of the file, no explanation |
| `security` | Security audit — lists issues with severity |
| `ask` | Free-form question with no file target |

### Phase 2.1 — Developer operations (deferred)

| Operation | Description |
|-----------|-------------|
| `review` | Code review — bugs, style, and correctness issues |
| `document` | File returned with docstrings and comments added |
| `test` | Suggested test cases for the code |
| `optimise` | Refactored version focused on performance and efficiency |
| `diff` | Unified diff of suggested changes (pairs with `fix`) |

### Deferred to Phase 3

`diagnose/<service>` — requires live system state collection (systemctl, journalctl, ps). Fits better alongside Phase 3's deep system integration (eBPF, systemd supervision).

## Architecture

```
kernel VFS
    │  cat /ai/explain/etc/nginx/nginx.conf
    ▼
spai-fuse  (new binary)
    │  1. parse path → op="explain", target="/etc/nginx/nginx.conf"
    │  2. read target file from disk
    │  3. resolve timeout (SPAI_TIMEOUT env > config > 60s default)
    │  4. send FuseRequest{Op, FileName, Content} via Unix socket
    │  5. block until FuseResponse (or timeout)
    │  6. return response bytes to kernel
    ▼
spaid  (existing daemon — one new handler added)
    │  build operation-specific prompt
    │  route to AI via existing router (cloud or local)
    │  collect full streamed response
    │  return FuseResponse
    ▼
AI provider (unchanged)
```

**`spai-fuse`** is a separate binary. `spaid` gets one new handler (`onFuse`) and one new wire type. No changes to routing, sessions, permissions, agent loop, or LLM manager.

## Virtual Filesystem Layout

```
/ai/
├── explain/          ← mirrors real filesystem
│   └── etc/
│       └── nginx/
│           └── nginx.conf
├── summarise/
├── fix/
├── security/
└── ask/              ← no filesystem mirror; path IS the question
    └── what is using port 8080
```

**Path parsing:**

```
/ai/explain/etc/nginx/nginx.conf
     ↑       ↑
     op      real path (prepend "/" to restore)

/ai/ask/what process is using port 8080
     ↑   ↑
     op  question string (decoded from path, spaces preserved)
```

**Directory reads** (`ls /ai/explain/etc/`) return the real directory listing — tab-completion works naturally. No AI call is made. `ls /ai/ask/` returns an empty directory — there is no real filesystem to mirror for free-form questions.

**File size limit:** Files over 512 KB are truncated before sending, with `[truncated at 512KB]` appended.

## Session Handling

FUSE reads are **always ephemeral**. Each `cat` triggers a one-shot AI call — no session history is written, no session is read. The AI has exactly the file content as context, nothing more.

### Session options considered

Three options were evaluated:

**Option A — Ephemeral (chosen):** Each FUSE read is a one-shot request. No session touched. Simple, predictable, correct for filesystem semantics. The file content itself is the full context.

**Option B — Shell session aware:** FUSE reads join the active pinned/default shell session. Rejected because FUSE is a kernel-level mechanism with no shell environment — `$SPAI_SESSION_ID` is not in scope at read time. Mixing OS filesystem operations with interactive shell sessions creates noisy history and conflates two distinct interaction models.

**Option C — Dedicated FUSE session (natural upgrade path):** All FUSE reads accumulate in a long-lived `"fuse"` session. Reads across the same session can reference each other. More complex but reasonable if cross-read context becomes valuable. Revisit in a future phase.

## Wire Protocol

New additions to `internal/protocol/protocol.go`:

```go
type FuseRequest struct {
    Op       string // "explain" | "summarise" | "fix" | "security" | "ask"
    FileName string // real path for file ops; question string for "ask"
    Content  string // file content (empty for "ask")
}

type FuseResponse struct {
    Output string // full AI response text
    Error  string // non-empty on failure or timeout
}
```

`Request.Fuse *FuseRequest` added to the existing `Request` struct alongside `Agent`, `Session`, and `LLM`.

### Prompt templates (in `onFuse` handler)

| Op | Prompt |
|----|--------|
| `explain` | `Explain what this file does ({{FileName}}):\n{{Content}}` |
| `summarise` | `Summarise this file ({{FileName}}) in 3–5 sentences:\n{{Content}}` |
| `fix` | `Return a corrected version of this file ({{FileName}}). Output only the fixed file content, no explanation:\n{{Content}}` |
| `security` | `Audit this file ({{FileName}}) for security issues. List each issue with its severity:\n{{Content}}` |
| `ask` | `Answer this system question: {{FileName}}` |

`onFuse` collects the full streamed AI response before returning — no mid-stream writes, since FUSE `Read` must return complete content.

## Timeout

```
Priority: SPAI_TIMEOUT env var > [fuse] timeout_seconds in config > 60s default
```

`spai-fuse` reads the calling process's environment via `/proc/<pid>/environ` (PID provided by `fuse.Context.Pid`) to check for `SPAI_TIMEOUT`.

```bash
# Per-command override
SPAI_TIMEOUT=120 cat /ai/fix/etc/nginx/nginx.conf
SPAI_TIMEOUT=0   cat /ai/summarise/var/log/syslog   # no timeout
```

Config:

```toml
[fuse]
auto_mount = true
mountpoint = "/ai"
timeout_seconds = 60    # 0 = no timeout
```

## Mount Management

### systemd unit — `spai-fuse.service`

```ini
[Unit]
Description=spaiSH FUSE filesystem
After=spaid.service
Requires=spaid.service

[Service]
ExecStart=%h/.local/bin/spai-fuse --mountpoint=/ai
ExecStop=fusermount3 -u /ai
Restart=on-failure

[Install]
WantedBy=default.target
```

Auto-mount is on by default. Disable via config:

```toml
[fuse]
auto_mount = false
```

### Manual control

```bash
spai mount              # start spai-fuse, mount /ai
spai mount --at /myai   # custom mountpoint
spai unmount            # fusermount3 -u /ai
spai unmount --at /myai
```

Manual `spai mount`/`unmount` always works regardless of `auto_mount`.

### install.sh additions

```bash
sudo mkdir -p /ai
systemctl --user enable spai-fuse.service
systemctl --user start spai-fuse.service
```

### uninstall.sh additions

```bash
systemctl --user stop spai-fuse.service
systemctl --user disable spai-fuse.service
sudo rmdir /ai
```

## Error Handling

All errors are returned as readable file content — `cat` always produces something useful.

| Situation | Output |
|-----------|--------|
| AI timeout | `[spaiSH error: request timed out after 60s — try SPAI_TIMEOUT=120]` |
| `spaid` not running | `[spaiSH error: daemon not running — run 'spai mount' or check 'systemctl --user status spaid']` |
| Target file not found | `[spaiSH error: file not found: /etc/nginx/nginx.conf]` |
| Target file unreadable | `[spaiSH error: permission denied: /etc/shadow]` |
| File over 512 KB | content truncated with `[truncated at 512KB]` appended |
| Unknown operation | `[spaiSH error: unknown operation 'foo' — valid: explain, summarise, fix, security, ask]` |
| AI provider unavailable | `[spaiSH error: no AI provider available — check spaid.toml or run 'spai llm status']` |

## Go Dependency

FUSE filesystem implementation uses `github.com/hanwen/go-fuse/v2` — the standard Go FUSE library, actively maintained, used in production systems.

## Success Criteria

A developer installs spaiSH on a fresh Linux machine. After installation, `/ai` is mounted automatically. Running:

```bash
cat /ai/explain/etc/nginx/nginx.conf
cat /ai/fix/etc/fstab
cat "/ai/ask/what is eating my disk"
```

returns useful AI output within the configured timeout. `spai unmount` cleanly removes the mount. `spai mount` restores it.
